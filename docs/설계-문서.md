## 1. Electron 프로세스 분리

### Main Process (Node.js)
백엔드 역할. UI 없음.

담당:
- Provider 관리 (CLI 서브프로세스, API 호출, 로컬 LLM 연결)
- 파일 시스템 접근 및 권한 제어
- SQLite 데이터베이스 (메모리 시스템)
- 대화 엔진 (턴 관리, 합의 흐름)
- 대화 로그 저장 (md 파일)

### Renderer Process (React)
프론트엔드 역할. 화면만 담당.

담당:
- 대화 UI (스트리밍 표시, 애니메이션)
- 설정 화면
- 파일/폴더 선택 다이얼로그
- 참여자 관리 UI
- 권한 설정 UI

### 원칙
- Renderer는 절대 파일 시스템이나 서브프로세스에 직접 접근하지 않음
- 모든 데이터는 IPC를 통해 Main에 요청

## 2. IPC 통신 설계

Electron의 `ipcMain` / `ipcRenderer` 사용.
TypeScript로 채널과 페이로드를 타입 정의하여 안전성 확보.

### 공통 메타 필드

모든 IPC 메시지는 공통 메타 정보를 포함한다:

```typescript
interface IpcMeta {
  requestId: string;         // 요청 추적 ID
  conversationId?: string;   // 대화 세션 ID
  sequence?: number;         // 스트리밍 유실/중복 감지용 순번
  schemaVersion: number;     // 업데이트 시 버전 불일치 대응
  timestamp: number;         // 메시지 생성 시각
}
```

### Typed IPC Wrapper

IPC 호출은 `IpcChannelMap` 기반 typed wrapper를 사용하여 타입 안전성을 확보한다:

```typescript
type IpcChannelMap = {
  'chat:send': {
    request: { content: string; attachments?: FilePath[] };
    response: void;
  };
  'provider:list': {
    request: void;
    response: Provider[];
  };
  'consensus:respond': {
    request: { action: 'approve' | 'reject' | 'revise'; comment?: string };
    response: void;
  };
  // ... 각 채널별 정의
};

function typedInvoke<C extends keyof IpcChannelMap>(
  channel: C,
  data: IpcChannelMap[C]['request'],
  meta?: Partial<IpcMeta>
): Promise<IpcChannelMap[C]['response']>;
```

**런타임 검증**:
- 런타임 스키마 검증(zod)은 개발 모드에서만 활성화한다.
- 프로덕션에서는 TypeScript 컴파일타임 검증에 의존한다.

### 채널 목록

모든 채널은 `IpcChannelMap`에 정의되며, 다음을 포함한다:

```typescript
// ===== 대화 =====

// Renderer → Main: 사용자 메시지 전송
'chat:send'        { content: string, attachments?: FilePath[] }

// Main → Renderer: AI 응답 스트리밍 (토큰 단위)
'chat:stream'       { participantId: string, token: string, done: boolean, sequence: number }

// Main → Renderer: 대화 상태 변경
'chat:status'       { participantId: string, status: 'thinking' | 'waiting' | 'speaking' | 'idle' }

// Renderer → Main: 대화 제어
'chat:pause'        {}
'chat:resume'       {}
'chat:stop'         {}

// Renderer → Main: 라운드 설정 변경
'chat:set-rounds'   { rounds: number | 'unlimited' }

// ===== 합의 (작업 모드) =====

// Main → Renderer: 합의 제안 도착
'consensus:proposal'  { proposal: string, diff?: DiffView, round: number }

// Renderer → Main: 사용자 승인/거부
'consensus:respond'   { action: 'approve' | 'reject' | 'revise', comment?: string }

// ===== Provider 관리 =====

// Renderer → Main: Provider CRUD
'provider:list'       {}
'provider:add'        { type: 'api' | 'cli' | 'local', config: ProviderConfig }
'provider:update'     { id: string, config: Partial<ProviderConfig> }
'provider:remove'     { id: string }
'provider:validate'   { id: string }

// Main → Renderer: Provider 상태
'provider:status'     { id: string, status: 'ready' | 'error' | 'not-installed', message?: string }

// ===== 파일/권한 =====

// Renderer → Main: 폴더 선택
'file:select-folder'  {}

// Main → Renderer: 선택 결과
'file:folder-selected' { path: string }

// Renderer → Main: 권한 설정
'file:set-permissions' { folderId: string, permissions: PermissionSet[] }

// ===== 메모리 =====

// Renderer → Main: 메시지 핀
'memory:pin'          { messageId: string, topic: 'technical' | 'decisions' | 'preferences' | 'context' }

// Renderer → Main: 메모리 검색
'memory:search'       { query: string }

// Main → Renderer: 검색 결과
'memory:results'      { memories: MemoryNode[] }
```

## 3. Provider 인터페이스 설계

모든 AI provider가 공통 인터페이스를 구현한다.

### 핵심 타입 정의

```typescript
type ProviderCapability =
  | 'streaming' | 'resume' | 'tools'
  | 'web-search' | 'json-mode' | 'multimodal' | 'code-execution';

type ProviderConfig =
  | { type: 'api'; endpoint: string; apiKeyRef: string; model: string }
  | { type: 'cli'; command: string; sessionMode: 'persistent' | 'per-turn'; model: string }
  | { type: 'local'; baseUrl: string; model: string };

interface ProviderInterface {
  // 식별
  id: string;
  type: 'api' | 'cli' | 'local';
  displayName: string;
  model: string;
  capabilities: Set<ProviderCapability>;
  config: ProviderConfig;

  // 라이프사이클
  warmup(): Promise<void>;          // 참여자 선택 시 호출 (프리커넥트)
  cooldown(): Promise<void>;        // 참여자 해제 시 호출 (리소스 정리)
  validateConnection(): Promise<boolean>;  // 연결 확인
  ping(): Promise<boolean>;         // 경량 헬스 체크

  // 핵심: 스트리밍 응답 생성
  streamCompletion(
    messages: Message[],
    persona: string,
    options?: CompletionOptions,
    signal?: AbortSignal
  ): AsyncGenerator<string>;

  // 메시지 변환 (Provider별 형식 적응)
  adaptMessages(messages: Message[]): ProviderSpecificMessage[];

  // 상태
  isReady(): boolean;
  getStatus(): ProviderStatus;
  onStatusChange(callback: (status: ProviderStatus) => void): void;
}

interface Message {
  role: 'user' | 'assistant' | 'system';
  content: string | ContentBlock[];  // 멀티모달 지원
  name?: string;                      // 멀티파티 대화에서 발화자 이름
  metadata?: Record<string, unknown>;
}

interface ContentBlock {
  type: 'text' | 'image' | 'tool_use' | 'tool_result';
  data: unknown;
}

interface CompletionOptions {
  temperature?: number;
  maxTokens?: number;
  tools?: ToolDefinition[];
  [key: string]: unknown;
}

type ProviderStatus = 'ready' | 'warming-up' | 'busy' | 'error' | 'not-installed';
```

**설계 원칙**:
- Provider 분기는 `type` 하드코딩이 아니라 `capabilities` 기반으로 수행한다.
- 예: 스트리밍 지원 여부는 `provider.capabilities.has('streaming')`으로 확인

### Provider별 구현 전략

#### API Provider (OpenAI, Anthropic, Google 등)
```
구현: HTTP 클라이언트로 직접 API 호출
스트리밍: SSE 파싱하여 토큰 단위 yield
warmup: HTTP 커넥션 프리커넥트 (keep-alive)
장점: Node.js에서 네이티브, 가장 안정적
```

#### CLI Provider (Gemini CLI, Claude Code, Codex CLI)

**공통 구조**:
- 공통 `CliProvider` 베이스 클래스 + config-driven 구조
- 각 CLI는 config 파일로 파서/세션 전략 정의

```typescript
interface CliProviderConfig {
  command: string;                    // 실행 명령어
  args: string[];                     // 기본 인자
  inputFormat: 'stdin-json' | 'args' | 'pipe';
  outputFormat: 'stream-json' | 'jsonl' | 'raw-stdout';
  sessionStrategy: 'persistent' | 'per-turn';
  hangTimeout: { first: number; subsequent: number };  // 응답 대기 시간
  outputParser?: (raw: string) => string;              // 커스텀 파서
}
```

**CLI별 config 예시**:

```typescript
// Gemini CLI
const geminiConfig: CliProviderConfig = {
  command: 'gemini',
  args: ['--headless', '--format', 'json'],
  inputFormat: 'args',
  outputFormat: 'raw-stdout',
  sessionStrategy: 'per-turn',  // 매 턴 subprocess 생성
  hangTimeout: { first: 30000, subsequent: 10000 },
  outputParser: (raw) => {
    // JSON 출력 시 불필요한 텍스트 정제 로직
    const jsonStart = raw.indexOf('{');
    return jsonStart >= 0 ? raw.slice(jsonStart) : raw;
  }
};
// warmup: --resume <session_id>로 세션 재사용

// Claude Code
const claudeConfig: CliProviderConfig = {
  command: 'claude',
  args: ['--input-format', 'stream-json', '--output-format', 'stream-json'],
  inputFormat: 'stdin-json',
  outputFormat: 'stream-json',
  sessionStrategy: 'persistent',  // persistent process 유지
  hangTimeout: { first: 60000, subsequent: 30000 },
};
// warmup: 프로세스 미리 생성

// Codex CLI
const codexConfig: CliProviderConfig = {
  command: 'codex',
  args: ['exec', '--json'],
  inputFormat: 'args',
  outputFormat: 'jsonl',
  sessionStrategy: 'per-turn',
  hangTimeout: { first: 45000, subsequent: 15000 },
};
// warmup: codex exec --ephemeral 빈 호출로 인증/모델 로딩 확인
// 모델: gpt-5.3-codex (기본), --model 플래그로 선택
// 웹 검색 내장 (web_search 도구)
```

#### Local Provider (Ollama 등)
```
구현: Ollama HTTP API 호출 (localhost:11434)
스트리밍: /api/chat 엔드포인트의 스트리밍 응답 파싱
warmup: 모델 로딩 트리거 (/api/generate로 빈 호출)
장점: 기본 스트리밍 지원, 가장 빠른 응답
```

### Provider Registry + Model Catalog

Provider 카탈로그는 런타임 데이터(DB 또는 JSON)로 관리한다:
- Phase 0: `version` 필드 예약
- Phase 1+: 카탈로그 업데이트 로직과 함께 버전 추적 활성화

### 4-2. 대화 흐름

#### 일반 대화

```
사용자 메시지 입력
    ↓
[컨텍스트 조립]
  시스템 프롬프트 (페르소나)
  + 메모리에서 관련 기억 검색
  + 최근 대화 히스토리
  + 현재 메시지
    ↓
[다음 AI 결정] (라운드 로빈 또는 순서)
    ↓
[AI 응답 생성] → 스트리밍으로 Renderer에 전달
    ↓
[2초 대기] → "..." 애니메이션
  ├── 사용자 끼어들면 → 대기 취소, 사용자 메시지 처리
  └── 2초 지나면 → 다음 AI 턴
    ↓
[라운드 체크]
  ├── 라운드 남음 → 다음 AI 턴
  └── 라운드 끝 → 대화 종료 (사용자가 추가 입력 가능)
```

#### 작업 대화 (Arena, 합의 기반)

```typescript
type ConsensusState =
  | 'DISCUSSING' | 'SYNTHESIZING' | 'VOTING'
  | 'AWAITING_USER' | 'APPLYING' | 'DONE' | 'FAILED';

type ConsensusEvent =
  | 'ROUND_DONE' | 'ALL_AGREE' | 'DISAGREE'
  | 'USER_APPROVE' | 'USER_REVISE' | 'USER_REJECT'
  | 'TIMEOUT' | 'ERROR';

interface ConsensusConfig {
  maxRetries: number;                  // 동의 실패 루프 상한
  phaseTimeout: number;                // 단계별 타임아웃
  aggregatorStrategy: 'strongest' | 'last-speaker' | 'designated' | 'round-robin';
}
```

**합의 흐름 (ConsensusStateMachine)**:

```
사용자: 파일 + 주제 전달
    ↓
[DISCUSSING 상태] — 일반 대화와 동일하게 진행
  라운드만큼 AI끼리 토론
  사용자도 언제든 개입 가능
    ↓
ROUND_DONE 이벤트
    ↓
[SYNTHESIZING 상태]
  aggregatorStrategy에 따라 취합자 결정:
    - strongest: 가장 강력한 모델
    - last-speaker: 마지막 발언 AI
    - designated: 사전 지정된 AI
    - round-robin: 순환 배정
  취합 문서 생성 → .arena/workspace/proposals/에 저장
    ↓
[VOTING 상태]
  각 AI에게 취합 문서 전달 + "동의하는가? 코멘트를 달라"
  AI 응답 분석:
    ├── 전원 "동의" (ALL_AGREE) → [AWAITING_USER 상태]로
    ├── 코멘트/수정 제안 (DISAGREE) → 수정 후 재VOTING (maxRetries 체크)
    └── 재시도 한계 → [FAILED 상태]
    ↓
[AWAITING_USER 상태]
  Renderer에 consensus:proposal 전송
  diff 미리보기 표시
  사용자 선택:
    ├── 승인 (USER_APPROVE) → [APPLYING 상태]
    ├── 수정요청 (USER_REVISE) → [DISCUSSING 상태]로 복귀
    └── 거부 (USER_REJECT) → [FAILED 상태]
    ↓
[APPLYING 상태]
  ExecutionService를 통해 변경사항 실제 반영
  성공 → [DONE 상태]
  실패 → rollback + [FAILED 상태]
```

**상태 스냅샷 저장 규칙**:
- 전이 이벤트마다 즉시 저장
- 5초 주기 체크포인트 (장시간 단계용)
- 종료 시 flush

### 4-3. 멀티파티 메시지 변환

기존 AI Chat Arena의 핵심 로직 유지:

```
AI-X의 관점에서 메시지 변환:
  - X의 발언 → role: "assistant"
  - 다른 모든 발언 → role: "user", content에 "[이름]: 내용" 접두사
  - 연속된 user 메시지는 병합 (API 호환성)
```

각 Provider의 `adaptMessages()` 메서드가 이 로직을 처리한다.

### 4-4. 합의/작업 모드 세부 규칙

> 원래 `conversation-and-task-mode-design.md`에 별도 기술된 합의 사항을 흡수.

#### AI 투표 스키마 (v1)

```json
{
  "decision_schema_version": "v1",
  "decision": "AGREE | DISAGREE | BLOCK",
  "block_reason_type": "security | data_loss | spec_conflict | unknown",
  "reason": "string"
}
```

- `block_reason_type`은 `decision=BLOCK`일 때만 필수
- `reason`은 항상 필수
- 파싱 실패 시 최대 2회 재시도, 그래도 실패하면 `ABSTAIN` 처리 후 사용자 알림

#### 투표 규칙

- **참가자 수 규칙**: 2인 → 만장일치, 3인+ → 과반수 허용 (소수 의견 보존)
- **최소 유효 투표**: `ceil(participant_count / 2)` (2→2, 3→2, 4→3, 5→3)
- **ABSTAIN**: 분모에서 제외. 유효 투표 < 최소 유효 투표 시 합의 무효
- **BLOCK**: `security`/`data_loss` → 하드 블록 (즉시 중단), `spec_conflict`/`unknown` → 사용자 확인 필요
- **투표 무효화**: 사용자 새 메시지 전송 또는 `proposal_hash` 변경 시 전체 투표 무효

#### 취합자 (Facilitator)

- 시스템이 기본 취합자 AI를 지정
- 사용자가 런타임에 취합자 변경 가능
- 합의 성공 후 취합자가 최종 합의 문서 작성

#### 실행/리뷰 정책

- 승인된 계획 단위로 일괄 실행
- Diff 표시/리뷰는 파일별
- 실패 시 부분 적용 불허
- 실행/리뷰 실패 → 항상 사용자 결정 경유: 재시도 / 중단 / 실행자 AI 재배정
- 자율 무한 자기 수정 루프 금지

#### 가시성/감사 정책

- 읽기 작업은 채팅에 표시하지 않음
- 변경 사항은 기본적으로 diff 카드로 표시
- 컨텍스트 스니펫은 on-demand (인접 행 제한)
- 내부 감사 로그는 읽기 추적 및 근거 매핑 유지

#### MCP 결정

- 현재 아키텍처에서 MCP 불필요
- 앱이 중앙 오케스트레이터 역할 (수집 → 판단 → 컨텍스트 주입)
- 외부 도구 탐색 상호운용이 필요해질 경우에만 MCP 어댑터 추가

## 5. 실행 안전 계층

AI가 제안한 파일 변경이나 명령 실행은 반드시 `ExecutionService`를 통과한다.

### 핵심 인터페이스

```typescript
interface ExecutionService {
  readFile(path: string, aiId: string): Promise<string>;
  writeFile(path: string, content: string, aiId: string): Promise<void>;
  listDir(path: string, aiId: string): Promise<string[]>;
  applyPatch(patch: Patch, aiId: string): Promise<ApplyResult>;
  runCommand(request: CommandRequest, aiId: string): Promise<CommandResult>;
}

interface CommandRequest {
  command: string;    // 실행 파일 경로 또는 명령어
  args: string[];     // 인자 배열
  cwd: string;        // 작업 디렉토리
}

interface CommandPolicy {
  allowedCommands: string[];     // 허용 명령어 화이트리스트
  blockedPatterns: string[];     // 위험 패턴 블랙리스트
  maxExecutionTimeMs: number;    // 최대 실행 시간
  maxOutputBytes: number;        // 최대 출력 크기
}

interface AuditEntry {
  operationId: string;
  aiId: string;
  action: string;           // 'read' | 'write' | 'execute' | 'apply-patch'
  targetPath: string;
  timestamp: number;
  result: 'success' | 'denied' | 'failed';
  rollbackable: boolean;
}
```

### 실행 파이프라인

```
1. [Dry-run diff]
   AI 제안 → ExecutionService.applyPatch(patch, aiId, { dryRun: true })
   → diff 생성, 실제 반영 안 함

2. [사용자 승인]
   Renderer에 diff 표시 → consensus:proposal
   사용자: 승인 / 수정요청 / 거부

3. [Atomic apply]
   승인 시 → ExecutionService.applyPatch(patch, aiId, { dryRun: false })
   → 트랜잭션 방식으로 전체 또는 전무 반영

4. [Rollback on failure]
   실패 시 → 변경 전 상태로 복원
   → AuditEntry에 실패 기록
```

### 명령 실행 안전

`runCommand`는 구조화된 요청으로만 처리하며, 셸을 경유하지 않는다:

```typescript
// ❌ 위험: 셸 인젝션 가능
child_process.exec(`${command} ${args.join(' ')}`);

// ✅ 안전: 구조화된 실행
child_process.execFile(request.command, request.args, {
  cwd: request.cwd,
  timeout: policy.maxExecutionTimeMs,
  maxBuffer: policy.maxOutputBytes,
  shell: false,  // 셸 미경유
});
```

**CommandPolicy 적용**:
- 화이트리스트에 없는 명령어는 차단
- 블랙리스트 패턴 (예: `rm -rf /`, `chmod 777`) 매칭 시 차단
- 타임아웃/출력 크기 제한 초과 시 종료

### 감사 로그

모든 실행 시도는 `AuditEntry`로 기록된다:
- 성공/실패 여부
- 실행자(aiId), 대상 경로, 시각
- rollbackable 플래그 (롤백 가능 여부)

## 7. 메모리 시스템 설계

### 7-1. 데이터베이스 스키마

SQLite + FTS5, bara_system 기반 확장

```sql
-- 지식 노드 (핵심 기억 단위)
CREATE TABLE knowledge_nodes (
  id          TEXT PRIMARY KEY,
  content     TEXT NOT NULL,           -- "Python 3.12로 결정"
  embedding   BLOB,                    -- 벡터 임베딩
  node_type   TEXT NOT NULL,           -- 'fact', 'decision', 'preference', 'insight'
  topic       TEXT NOT NULL,           -- 'technical', 'decisions', 'preferences', 'context'
  importance  REAL DEFAULT 0.5,        -- 0~1 중요도
  source      TEXT,                    -- 'auto' | 'pin' | 'reflection'
  pinned      INTEGER DEFAULT 0,       -- 사용자 핀 여부
  conversation_id TEXT,                -- 출처 대화 ID
  message_id  TEXT,                    -- 출처 메시지 ID
  last_accessed DATETIME,
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP,

  -- 운영 컬럼 (Phase 3-b)
  embedding_version TEXT,              -- 임베딩 모델 버전
  extractor_version TEXT,              -- 추출기 버전
  source_hash TEXT,                    -- 원본 내용 해시 (중복 감지)
  dedupe_key TEXT,                     -- 중복 제거 키
  deleted_at DATETIME                  -- soft delete
);

-- FTS5 전문검색 인덱스
CREATE VIRTUAL TABLE knowledge_fts USING fts5(
  content,
  content=knowledge_nodes,
  content_rowid=rowid,
  tokenize='unicode61'
);

-- 지식 그래프 에지 (노드 간 관계)
CREATE TABLE knowledge_edges (
  id              TEXT PRIMARY KEY,
  source_node_id  TEXT REFERENCES knowledge_nodes(id),
  target_node_id  TEXT REFERENCES knowledge_nodes(id),
  relation_type   TEXT NOT NULL,       -- 'related_to', 'contradicts', 'supersedes', 'depends_on'
  weight          REAL DEFAULT 1.0,
  created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 대화 기록
CREATE TABLE conversations (
  id          TEXT PRIMARY KEY,
  title       TEXT,
  mode        TEXT NOT NULL,           -- 'free' | 'work'
  participants TEXT NOT NULL,          -- JSON array of participant IDs
  folder_path TEXT,                    -- 작업 모드 시 폴더 경로
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 메시지 기록 (분기 지원)
CREATE TABLE messages (
  id              TEXT PRIMARY KEY,
  conversation_id TEXT REFERENCES conversations(id),
  participant_id  TEXT,                -- AI id 또는 'user'
  content         TEXT NOT NULL,
  role            TEXT NOT NULL,       -- 'user' | 'assistant' | 'system'
  pinned          INTEGER DEFAULT 0,
  pin_topic       TEXT,                -- 핀 시 토픽
  response_time_ms INTEGER,            -- 응답 소요 시간
  token_count     INTEGER,             -- 토큰 수
  created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,

  -- 분기 지원 필드 (Phase 0부터 포함)
  parent_message_id TEXT,              -- 부모 메시지 ID
  branch_id TEXT,                      -- 분기 식별자
  branch_root_message_id TEXT          -- 분기 시작점
);

-- Provider 설정
CREATE TABLE providers (
  id          TEXT PRIMARY KEY,
  type        TEXT NOT NULL,           -- 'api' | 'cli' | 'local'
  name        TEXT NOT NULL,           -- 사용자 지정 이름
  model       TEXT NOT NULL,
  persona     TEXT,                    -- 시스템 프롬프트
  config      TEXT NOT NULL,           -- JSON (API key, endpoint 등)
  permissions TEXT,                    -- JSON (기본 권한)
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 7-2. 메모리 모듈 구조

```
memory/
├── facade.ts           통합 파사드 (외부 인터페이스)
├── retriever.ts        하이브리드 검색 (벡터 + FTS5 + 그래프)
├── extractor.ts        LLM/Regex 자동 추출 (대화에서 사실/결정 추출)
├── scorer.ts           Stanford 3-factor 스코어링
├── assembler.ts        토큰 예산 컨텍스트 조립
├── evolver.ts          메모리 진화 (병합/정리/중요도 재계산)
├── reflector.ts        반성 엔진 (인사이트 생성)
└── token-counter.ts    js-tiktoken 기반 정확한 토큰 카운팅
```

**토큰 카운팅**:
- `js-tiktoken` 기반 정확한 카운팅을 기본으로 한다.
- 모델별 토크나이저 사용 (gpt-4, claude-3 등)

### 7-3. 이중 경로 저장 흐름

```
경로 1: 자동 저장 (매 AI 응답 후)
  메시지 → extractor.ts
    ├── Regex 빠른 경로: "~로 결정", "~를 쓰자" 패턴
    └── LLM 추출: 주요 사실/결정/관계 구조화 (Phase 3-b)
  → knowledge_nodes에 저장 (source: 'auto')

경로 2: 수동 저장 (사용자 핀)
  사용자가 메시지 핀 + 토픽 선택
    ├── 새 기억이면 → knowledge_nodes에 저장 (source: 'pin')
    └── 이미 자동 추출된 기억이면 → importance 부스트 + pinned = 1

컨텍스트 조립 (매 AI 턴 전):
  assembler.ts가 토큰 예산 내에서 조립:
    시스템 프롬프트     15%
    관련 메모리         25%  ← retriever.ts + scorer.ts
    최근 대화 히스토리  50%
    응답 예약           10%
```

### 단계적 접근

**Phase 3-a**: FTS5 + 핀 + Regex 추출
- 외부 의존성 없음 (임베딩 모델 불필요)
- Regex 패턴으로 주요 결정/사실 추출
- FTS5 전문검색으로 관련 기억 검색

**Phase 3-b**: 임베딩/하이브리드/반성/진화
- 임베딩 벡터 추가
- 하이브리드 검색 (벡터 + FTS5)
- 반성 엔진으로 인사이트 자동 생성
- 메모리 진화 (중복 병합, 중요도 재계산)

## 11. 설계 결정 요약

| 결정 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | Electron + TypeScript | CLI 연동 네이티브, Taleson 경험 활용 |
| 프론트엔드 | React | 클로드 코드 최적, 생태계 최대 |
| DB | SQLite + FTS5 | 단일 파일, 전문검색 내장, Bara와 호환 |
| 상태관리 | Zustand (권장) | 가볍고 TypeScript 친화적 |
| IPC | Typed channels | 타입 안전한 Main ↔ Renderer 통신 |
| CLI 관리 | child_process | Node.js 네이티브, Python 대비 한 겹 적음 |
| 스트리밍 | AsyncGenerator | Provider 공통 인터페이스로 통일 |
| 메모리 | bara_system 기반 확장 | 검증된 구조 + 더 적극적 저장/활용 |
| 권한 | Main Process 검증 | Renderer에서 우회 불가 |
| 빌드 | electron-builder | 단일 바이너리 배포 |
| i18n | react-i18next | 키 기반 렌더링, CI 가드, 다국어 확장 대비 |
| 실행 계층 | ExecutionService | 화이트리스트 + atomic apply + 감사 로그 |
| 합의 엔진 | ConsensusStateMachine | 상태/이벤트/가드/타임아웃/스냅샷 |

### 12-3. 파일 시스템 보안

```
문제: AI가 path traversal (../../etc/passwd)로 허용 범위 밖 파일 접근

대응: Main Process에서 모든 파일 접근을 검증
```

```typescript
// files/permission-service.ts
function validateAccess(
  aiId: string,
  targetPath: string,
  action: 'read' | 'write' | 'execute'
): boolean {
  // 1. 경로 정규화
  const normalized = path.resolve(targetPath);

  // 2. 심볼릭 링크 추적 방지
  const realPath = fs.realpathSync(normalized);

  // 3. .arena/workspace는 전원 허용
  if (realPath.startsWith(workspacePath)) return true;

  // 4. 프로젝트 폴더 범위 체크
  if (!realPath.startsWith(projectFolderPath)) return false;

  // 5. AI별 권한 체크
  return checkPermission(aiId, action);
}
```

### 12-4. CLI 서브프로세스 보안

```
문제: AI가 CLI를 통해 임의 시스템 명령 실행 가능

대응: ExecutionService의 CommandPolicy를 통해 허용 명령 allowlist + 위험 패턴 차단
  - execute 권한이 있는 AI만 CLI 명령 가능
  - 작업 디렉토리(cwd)를 허용된 폴더로 강제 제한
  - 모든 실행 명령 로깅 (누가, 언제, 무엇을)
  - 구조화된 CommandRequest로 셸 인젝션 원천 차단
```

```typescript
// execution/execution-service.ts
async runCommand(request: CommandRequest, aiId: string): Promise<CommandResult> {
  // 1. CommandPolicy 검증
  if (!policy.allowedCommands.includes(request.command)) {
    throw new Error(`Command not allowed: ${request.command}`);
  }

  // 2. 위험 패턴 차단
  const fullCommand = `${request.command} ${request.args.join(' ')}`;
  for (const pattern of policy.blockedPatterns) {
    if (new RegExp(pattern).test(fullCommand)) {
      await auditLog.record({
        aiId,
        action: 'execute',
        targetPath: request.cwd,
        result: 'denied',
        reason: `Blocked pattern: ${pattern}`
      });
      throw new Error(`Blocked dangerous pattern: ${pattern}`);
    }
  }

  // 3. 구조화된 실행 (셸 미경유)
  const result = await execFile(request.command, request.args, {
    cwd: request.cwd,
    timeout: policy.maxExecutionTimeMs,
    maxBuffer: policy.maxOutputBytes,
    shell: false,
  });

  // 4. 감사 로그 기록
  await auditLog.record({
    aiId,
    action: 'execute',
    targetPath: request.cwd,
    result: 'success',
    command: fullCommand,
  });

  return result;
}
```

## 14. 운영 원칙

### 마이그레이션 정책

- **forward-only**: 롤백 스크립트 없음, 앞으로만 진행
- **idempotent**: 같은 마이그레이션 재실행 시 안전 (IF NOT EXISTS)
- **startup blocking on failure**: 마이그레이션 실패 시 앱 시작 차단
- **순번 기반 네이밍**: `001_initial_schema.sql`, `002_add_branch_fields.sql`

```typescript
// database/migrations/index.ts
async function runMigrations() {
  const applied = await getAppliedMigrations();
  const pending = migrations.filter(m => !applied.includes(m.id));

  for (const migration of pending) {
    try {
      await db.exec(migration.sql);
      await recordMigration(migration.id);
    } catch (err) {
      console.error(`Migration ${migration.id} failed:`, err);
      throw new Error(`Startup blocked: migration failure`);
    }
  }
}
```

### .arena/workspace git 자동관리

- **기본**: OFF (git 추적 안 함)

- 자동 커밋/푸시는 하지 않음 (사용자가 직접 관리)
