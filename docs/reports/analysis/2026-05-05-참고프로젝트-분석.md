## 참고 가치가 큰 설계

Symphony의 `WORKFLOW.md`는 단순 설정 파일이 아니라 “작업 수행 계약서”에 가깝다.

- front matter: tracker, polling, workspace, hooks, agent, codex runtime 설정

- 사용자 노출 설정이 아니라 내부 실행 policy 문서로 활용

주의:

- Symphony의 Linear/PR 중심 규칙은 Rolestra에 그대로 맞지 않는다.
- Rolestra는 UI 중심 앱이므로 workflow 파일을 사용자에게 직접 편집시키기보다, preset/template로 관리하는 편이 자연스럽다.

### 5. 관찰 가능성, snapshot, token accounting

Symphony는 운영자가 현재 상태를 볼 수 있게 running/retrying/token/rate-limit 중심 snapshot을 만든다.

좋은 점:

- dashboard는 correctness에 관여하지 않는 순수 관찰 표면

Rolestra 적용 후보:

효과:

- “누가 지금 책임자인가”가 명확해짐
- handoff history를 요약해서 넘길지, 전체를 넘길지 정책화 가능
- 부서 간 업무 이동을 audit log로 남기기 쉬움

### 3. Runner loop의 NextStep 모델

`src/agents/run_internal/run_steps.py`는 모델 응답을 곧바로 처리하지 않고 `ProcessedResponse`로 분류한 뒤 다음 단계로 넘긴다.

대표 next step:

- final output
- handoff
- run again
- interruption

Rolestra 적용 후보:

- `MeetingOrchestrator`
- `TurnManager`
- `ConsensusStateMachine`
- provider response parser
- `ExecutionService` approval boundary

현재 Rolestra의 회의/합의 상태 머신 위에 “한 턴의 실행 결과”를 별도 next-step union으로 두면 좋다. 모델 출력, tool call, approval interruption, handoff를 같은 문자열 처리 루프에 섞지 않을 수 있다.

### 4. Human-in-the-loop와 durable RunState

`docs/human_in_the_loop.md`와 `src/agents/run_state.py`는 Rolestra의 `ApprovalService`에 매우 유용하다.

핵심 흐름:

- 민감한 tool call이 approval 필요 여부를 판단
- approval이 없으면 run이 interruption 상태로 멈춤
- 현재 실행 상태를 `RunState`로 직렬화
- 사용자가 승인/거절하면 같은 top-level agent를 재개
- nested agent/tool/handoff 안에서 발생한 approval도 바깥 run에서 처리

Rolestra 적용 후보:

- `ApprovalService`
- approval inbox
- pending approval DB schema
- app restart 후 승인 대기 작업 복구
- `ExecutionService` dry-run/apply 경계

특히 참고할 점은 `RunState`에 schema version을 둔다는 것이다. Rolestra도 pending approval, queue item, meeting resume snapshot에 version marker를 두는 편이 안전하다.

### 5. Guardrail 적용 시점 분리

OpenAI Agents SDK는 guardrail을 세 가지로 분리한다.

- input guardrail: 최초 입력에 적용
- output guardrail: 최종 출력에 적용
- tool guardrail: custom function tool call마다 적용

Rolestra 적용 후보:

- `AutonomyGate`
- `CircuitBreaker`
- `ApprovalService`
- provider-independent safety policy
- 파일/명령 실행 전 blocking check

중요한 점은 tool guardrail을 tool 실행 직전에 두는 것이다. Rolestra에서는 사용자 입력 검열보다 “파일 쓰기, 명령 실행, 외부 provider 호출, MCP tool 호출 직전”의 guardrail이 더 중요하다.

### 6. Streaming event taxonomy

대표 event:

- tool output

- reasoning item
- approval requested/response
- mcp list tools

Rolestra 적용 후보:

Rolestra도 provider별 raw stream을 그대로 UI에 흘리기보다, 공통 semantic event로 normalize하는 레이어를 두면 timeline/approval/log가 훨씬 안정적이다.

### 7. Session/history 전략

`docs/sessions/index.md`의 핵심은 conversation state 전략을 섞지 않는 것이다.

## Symphony와의 비교

Symphony는 “오래 도는 자율 작업 운영” 참고에 강하다. 반면 `openai-agents-python`은 “에이전트 런타임 추상화” 참고에 강하다.

둘을 함께 보면 Rolestra의 방향은 다음처럼 나누는 편이 좋다.

- Symphony에서 가져올 것: queue orchestration, retry/reconcile, workspace lifecycle, 운영 snapshot
- openai-agents-python에서 가져올 것: agent contract, handoff, tool/approval interruption, stream event, session/resume model

즉 Symphony는 운영 레이어, openai-agents-python은 실행 레이어 참고 프로젝트로 보는 것이 맞다.

## 권장 후속 작업

1. R12 설계 문서에 `AgentRuntimeNextStep` union 초안 추가
2. approval DB에 pending run snapshot/version marker를 둘지 검토
3. provider stream event를 `RawProviderEvent`와 `SemanticRunEvent`로 분리하는 ADR 작성
4. ArenaRoot/ExecutionService 위에 capability manifest 초안 작성
5. Codex CLI provider adapter를 experimental spike로 분리해서 검토

## 최종 판단

`openai-agents-python`은 Rolestra가 “AI 직원이 서로 넘겨주고, 도구를 쓰고, 승인 대기에서 멈췄다가, 같은 맥락으로 재개하고, 그 모든 과정을 UI timeline으로 설명하는 앱”이 되려면 매우 좋은 참고 프로젝트다.

우선순위는 Codex tool 포팅이 아니라 Runner loop, RunState, handoff, semantic stream event 모델을 TypeScript식으로 재해석하는 것이다.

## 6가지 invariant와 Rolestra 적용 지점

### 1. SSoT

의미:

- 시스템이 의존하는 truth는 단 하나의 canonical owner를 가져야 함
- cache/index/derived state가 drift하면 두 번째 truth를 만들지 말고 canonical truth에서 복구해야 함

Rolestra 적용 후보:

- SQLite transcript vs Zustand persisted state
- meeting snapshot vs channel message log
- provider-side conversation id vs Rolestra DB transcript
- theme token source vs generated CSS
- i18n source namespace vs UI 하드코딩 문자열
- approval pending state vs execution queue state

가장 중요한 적용점은 “provider resume state와 Rolestra DB transcript 중 무엇이 canonical인가”를 분명히 하는 것이다. provider-side state는 최적화일 수 있지만, Rolestra의 사용자-visible history와 audit truth는 DB 쪽이 canonical이어야 한다.

### 2. SoC / SRP

의미:

- 서로 다른 관심사는 분리된 경계에 두어야 함
- 하나의 boundary가 두 책임을 함께 지면 변경이 점점 어려워짐

Rolestra 적용 후보:

- provider adapter와 공통 provider registry 분리
- `MeetingOrchestrator`와 UI timeline rendering 분리
- `ExecutionService`와 approval decision UI 분리
- queue scheduling과 meeting turn execution 분리
- path guard와 command runner 분리
- persona/profile building과 runtime tool permission 분리

특히 Rolestra는 “AI 직원” 메타포가 강하므로 UX 개념이 runtime 책임과 섞이기 쉽다. 직원 card, 회의 timeline, provider turn execution, approval boundary가 한 서비스 안에 뭉치지 않게 유지하는 데 이 invariant가 유용하다.

### 3. Consistency

의미:

- 데이터, 상태, 표현이 시스템 전반에서 서로 모순되면 안 됨

Rolestra 적용 후보:

- queue item status와 meeting status의 불일치
- approval inbox에는 pending인데 execution은 already applied인 상태
- channel transcript와 meeting summary의 불일치
- provider cost audit log와 UI 비용 표시 불일치
- project-scoped Zustand persist와 main-process DB state 불일치

Symphony에서 본 runtime reconcile과 연결된다. Rolestra는 consistency violation을 “UI glitch”로 보지 말고, 복구 가능한 runtime invariant violation으로 다루는 편이 맞다.

### 4. Atomicity

의미:

- 변경은 완전히 끝나거나 롤백되어야 함
- half-written state를 외부에 노출하면 두 번째 truth가 생김

Rolestra 적용 후보:

- `ExecutionService` dry-run -> approval -> apply -> rollback
- SQLite migration chain
- file patch apply와 audit entry 기록
- approval decision과 execution state transition
- queue item claim/release
- generated theme/i18n/token artifacts

Rolestra의 기존 설계에서 `ExecutionService`는 이미 이 invariant와 잘 맞는다. 앞으로 자율 작업/부서 워크플로우가 커질수록 DB transaction과 file apply boundary를 더 명시적으로 묶어야 한다.

### 5. Idempotency

의미:

- 같은 요청을 반복해도 같은 결과로 수렴해야 함
- retry가 복구가 아니라 손상이 되면 안 됨

Rolestra 적용 후보:

- queue retry
- meeting resume
- provider stream event replay
- approval approve/reject 재전송
- migration 재실행
- notification 재발송
- cost usage event 재처리

Symphony의 retry/reconcile과 가장 잘 연결되는 invariant다. 자율 큐가 커지면 retry 가능한 작업과 retry하면 안 되는 작업을 타입 레벨 또는 DB state 레벨에서 분리해야 한다.

### 6. No Silent Fallback

의미:

- 실패를 조용히 숨기는 fallback, legacy path, shadow path는 금지
- availability를 위한 explicit failover는 가능하지만 observable해야 하고 canonical truth를 바꾸면 안 됨

Rolestra 적용 후보:

- provider fallback
- CLI provider failure 후 API provider로 자동 전환
- safeStorage 실패 후 plaintext 저장
- typed IPC validation 실패 후 unchecked invoke
- theme token 누락 후 임의 default 색상 사용
- translation key 누락 후 raw key 또는 English fallback
- path guard 실패 후 relaxed mode

Rolestra에서 가장 위험한 silent fallback은 보안/파일 실행 경계다. safeStorage, PathGuard, ApprovalService, ExecutionService에서 fallback은 기본적으로 fail-closed가 맞다. UI convenience fallback은 가능하더라도 diagnostics에 남겨야 한다.

## 참고 가치가 큰 설계

### 1. Canonical policy repo 패턴

`alex-core-invariants`는 정책 원문을 `README.md` 하나에만 둔다. 한국어 번역, skill, AGENTS 예시는 모두 파생물이다.

Rolestra 적용 후보:

- `docs/기능-정의서.md`
- `docs/설계-문서.md`
- `docs/코딩-규칙.md`
- `docs/완료-기준.md`
- `AGENTS.md`
- prompt/skill 문서

Rolestra는 문서가 많고 phase가 길어서 정책 drift 위험이 크다. “어떤 문서가 canonical이고, 어떤 문서는 derived/summary인가”를 문서마다 명확히 하는 방식이 유용하다.

### 2. Skill은 정책 전문을 복제하지 않는다

`SKILL.md`는 6 invariant 전문을 다시 쓰지 않고 `README.md wins`를 명시한다.

Rolestra 적용 후보:

- AGENTS.md와 role prompt 간 중복 축소
- setup/team/qa/review skill이 canonical docs를 링크만 하도록 정리
- 장기적으로 role prompt가 기능 정의/설계/완료 기준을 복제하지 않도록 관리

이는 Rolestra의 “여러 AI 직원” 구조에도 중요하다. 직원별 prompt가 각자 정책을 복제하면 시간이 지날수록 정책이 어긋난다.

### 3. 얕고 의견 있는 검증기

`checks/policy-scan.mjs`는 완벽한 정적 분석기가 아니다. 대신 다음 후보를 빠르게 찾는다.

- silent fallback
- legacy/shadow path
- dual-write/mirror-write
- catch-and-return

Rolestra 적용 후보:

- `npm run invariant-scan`
- PR/QA 전 선택 실행
- R12 자율 실행/approval/provider adapter 변경 시 reviewer checklist

처음부터 hard gate로 두기보다는, review candidate를 뽑는 보조 도구로 두는 편이 안전하다. false positive가 나올 수 있기 때문이다.

### 4. 예외 marker 방식

scanner는 다음 allow marker를 인식한다.

- `alex-allow-failover`
- `alex-allow-legacy`
- `alex-allow-dual-write`

Rolestra 적용 후보:

- `rolestra-allow-failover`
- `rolestra-allow-legacy`
- `rolestra-allow-dual-write`

단, marker만 두면 안 되고 이유와 ADR/이슈 링크를 요구하는 방식이 더 낫다.

예외 허용 조건:

- 왜 fallback/dual-write가 필요한지 설명
- canonical truth가 무엇인지 명시
- observable event/log가 있는지 확인
- 제거 예정 또는 bounded scope가 있는지 기록

### 5. pre-commit hook보다 CI/check script 우선

이 repo는 `.githooks/pre-commit`으로 `npm run check`를 실행한다. Rolestra에 그대로 가져오면 개발자 환경 의존성이 생길 수 있다.

Rolestra 적용 후보:

- pre-commit hook은 선택
- 기본은 `npm run check:docs` 또는 CI step
- 문서 canonical marker, generated artifact sync, invariant scan을 별도 script로 제공

## Rolestra에 적용한다면 우선순위

### 1순위: 코딩 규칙/리뷰 체크리스트에 6 invariant를 링크로 추가

대상:

- `docs/코딩-규칙.md`
- `docs/완료-기준.md`
- `AGENTS.md`
- QA/review report template

가져올 개념:

- SSoT
- SoC/SRP
- Consistency
- Atomicity
- Idempotency
- No Silent Fallback

주의:

- 전문을 복사하기보다 local reference path 또는 upstream link로 연결하는 편이 SSoT 원칙과 맞다.
- Rolestra 문서 안에는 “이 6 invariant 관점으로 검토한다” 정도만 두는 것이 좋다.

### 2순위: Rolestra-specific invariant scan 추가 검토

대상:

- provider adapters
- approval/execution/path guard
- queue/retry/reconcile
- migration scripts
- store persistence

가져올 개념:

- fallback/legacy/dual-write/catch-return 휴리스틱
- allow marker
- false positive를 review candidate로 취급

효과:

- silent fallback이 섞이는 순간을 빨리 잡을 수 있음
- AI-generated cleanup/refactor 이후 구조적 냄새를 빠르게 찾을 수 있음

### 3순위: canonical/derived 문서 관계 명시

대상:

- 기능 정의서
- 설계 문서
- ADR
- 구현 현황
- 디자인 README
- superpowers spec/plan

가져올 개념:

- canonical 문서 선언
- derived artifact에 source/hash marker
- drift check

효과:

- 장기 phase 문서 사이의 불일치가 줄어듦
- “어느 문서가 맞는가” 논쟁을 줄일 수 있음

### 4순위: fallback exception policy 수립

대상:

- provider failover
- safeStorage
- IPC validation
- i18n/theme default
- file/path execution

가져올 개념:

- silent fallback 금지
- explicit failover만 허용
- failover는 observable해야 함
- canonical truth를 바꾸면 안 됨

효과:

- 안전 경계가 조용히 완화되는 일을 방지
- 사용자에게 “왜 대체 경로로 갔는지” 설명 가능

## 가져오면 안 되는 부분

- 6 invariant 전문을 Rolestra 여러 문서에 복사하기
- `policy-scan.mjs`를 그대로 hard gate로 두기
- 한국어 번역본을 canonical 정책으로 삼기
- 모든 fallback을 무조건 금지하는 식으로 해석하기
- 작은 정책 repo의 단순성을 Rolestra 런타임 구조로 착각하기
- pre-commit hook을 모든 개발자에게 강제하기

## 세 참고 프로젝트의 역할 구분

지금까지 본 세 프로젝트를 나누면 다음과 같다.

- Symphony: 장기 자율 작업의 운영 레이어 참고
- openai-agents-python: 에이전트 실행/중단/재개/streaming 레이어 참고
- alex-core-invariants: 설계 리뷰/정책 SSoT/구조적 실패 방지 레이어 참고

이 셋은 서로 겹치기보다 보완적이다.

Rolestra에 적용할 때는 다음 순서가 자연스럽다.

1. `alex-core-invariants`로 core invariant 기준을 세운다.
2. `openai-agents-python`으로 실행 loop와 handoff/resume 모델을 설계한다.
3. Symphony로 queue/retry/reconcile/observability 운영 레이어를 보강한다.

## 권장 후속 작업

1. `docs/코딩-규칙.md`에 “Core invariant review lens” 섹션 추가 검토
2. fallback/dual-write/legacy path 예외 marker 정책 초안 작성
3. Rolestra 전용 `invariant-scan` script를 optional check로 실험
4. canonical 문서와 derived 문서 관계를 문서 인덱스에 명시
5. R12 자율 큐/approval/provider 작업부터 6 invariant checklist 적용

## 최종 판단

`alex-core-invariants`는 구현 참고 프로젝트라기보다 Rolestra가 앞으로 커질 때 구조적 실패를 줄이는 “리뷰 렌즈”로 가치가 있다.

가장 먼저 가져올 것은 스크립트가 아니라 원칙이다. 특히 Rolestra에는 `SSoT`, `Idempotency`, `No Silent Fallback`이 강하게 맞는다. provider resume, approval queue, execution rollback, safeStorage, path guard, i18n/theme fallback 같은 곳에서 이 원칙을 적용하면 장기 안정성에 도움이 된다.
