## 1. 결론 요약

세 참고 프로젝트에서 가져올 만한 핵심은 “기능 추가”보다 “런타임 안정성 모델의 강화”다. Rolestra는 이미 좋은 경계들을 갖고 있다. 특히 `ExecutionService`, `ApprovalService`, `QueueService`, `PermissionFlagBuilder`, `ConsensusStateMachine`은 방향이 맞다. 다만 현재 구조는 일부 흐름이 메모리 클로저, best-effort fallback, raw token stream, 단순 큐 상태에 의존한다. 자율 작업이 길어지고 회의가 복잡해질수록 이 부분이 결과물 품질과 안정성을 제한할 가능성이 크다.

품질 최우선 기준의 추천 반영 순서는 다음이다.

1. `RunState` / `RunStep` 영속 ledger를 도입해 회의, 승인, provider 호출, tool/파일 실행을 재시작 가능한 단위로 기록한다.
2. `AgentRuntimeNextStep` 같은 명시적 next-step union을 만들어 SSM, TurnManager, MeetingOrchestrator, ApprovalService 사이의 암묵적 제어 흐름을 줄인다.
3. raw token 중심 stream 위에 semantic run event 계층을 추가해 tool call, approval wait, handoff, retry, queue release를 UI와 감사 로그에 같은 의미로 노출한다.
4. Symphony식 큐 runtime layer를 붙여 `pending/in_progress/done`만이 아니라 `claimed/running/retrying/stalled/released`를 분리한다.
5. alex-core-invariants식 invariant catalog와 no-silent-fallback 정책을 문서, 테스트, 정적 스캔으로 묶는다.
6. capability manifest를 도입해 `PermissionService`, `PermissionFlagBuilder`, `ExecutionService`, CLI provider permission callback이 같은 권한 모델을 공유하게 한다.

가장 큰 기대 효과는 다음이다.

- 앱 재시작, provider 중단, 승인 지연, 큐 연속 실행에서 상태 유실이 줄어든다.
- 회의 결과의 품질이 올라간다. 모델 출력이 바로 상태 전이를 일으키는 대신, “다음에 무엇을 해야 하는가”가 구조화된다.
- 리뷰와 QA가 쉬워진다. “왜 이 파일이 바뀌었는가”, “어떤 승인이 어떤 실행을 열었는가”, “이 회의가 왜 멈췄는가”를 DB와 stream에서 추적할 수 있다.
- provider별 CLI fallback이나 session reset이 결과 품질에 미치는 영향을 관찰할 수 있다.

가장 큰 비용과 위험은 다음이다.

- 회의/승인/큐/stream/UI/DB migration이 동시에 닿는 큰 구조 변경이다.
- 기존 테스트의 snapshot, event order, UI optimistic update가 많이 흔들릴 수 있다.
- 보안 경계를 엄격하게 만들면 지금은 통과하던 명령이나 CLI 작업이 거부될 수 있다.
- provider session 정책을 고치면 토큰 비용, 답변 맥락, 재현성이 모두 바뀔 수 있다.

따라서 “빠른 패치”가 아니라 R12/R13급 구조 개선 phase로 다루는 것이 맞다.

- SSM 12상태가 이미 존재한다.

- `MeetingTurnExecutor`가 provider error, skipped member, DB persist error, token usage 기록을 한 곳에서 처리한다.

현재 한계:

- run-level 영속 ledger가 없어 provider call, approval wait, tool request, retry, handoff가 독립적인 복구 단위로 남지 않는다.

Provider interface는 `streamCompletion()` 중심이다. CLI provider는 persistent session과 per-turn session을 모두 지원하고, 실패 시 fallback을 수행한다. Stream은 renderer에 raw token과 turn done/error를 보낸다.

좋은 점:

- provider API, CLI, local provider를 한 interface로 묶었다.

현재 한계:

- semantic event가 부족하다. tool call, handoff, guardrail, approval wait, retry, queue release가 raw token/turn event와 별도로 구조화되어 있지 않다.

## 3. 반영 후보별 영향 분석

### A. openai-agents-python식 `RunState` / `RunStep` ledger

반영 내용:

- 회의 하나를 `run`으로 보고, speaker turn, model call, tool call, approval wait, queue claim, file apply, summary post를 `run_step`으로 기록한다.
- 각 step은 `pending/running/waiting_approval/succeeded/failed/cancelled/stalled` 같은 상태를 갖는다.
- step payload에는 provider id, model/session id, approval id, queue item id, meeting id, stream sequence, capability manifest hash를 연결한다.

장점:

- restart 이후에도 “어디서 멈췄는지”를 DB에서 복원할 수 있다.
- consensus approval 결정 후 minutes post/meeting finish가 누락되는 종류의 문제를 구조적으로 줄일 수 있다.
- 실패 분석이 쉬워진다. provider failure, user reject, queue cancel, file apply rollback이 같은 timeline에 남는다.
- R12-C2의 opinion tree, handoff, department workflow를 붙일 때 기반이 된다.

단점:

- DB migration, repository, service layer, stream bridge, renderer timeline, 테스트가 모두 필요하다.
- 기존 SSM snapshot과 역할이 겹친다. 중복 저장을 잘못 설계하면 “SSM은 DONE인데 RunStep은 waiting” 같은 모순이 생긴다.
- payload schema versioning이 필요하다. 오래된 run row를 새 코드가 읽을 수 있어야 한다.

사이드 이펙트:

- meeting stream event order가 바뀔 수 있다.
- 기존 테스트가 “turn token -> turn done -> DB message” 순서를 가정하고 있으면 깨질 수 있다.
- 승인 UI가 approval row뿐 아니라 연결된 run step 상태를 함께 보여야 한다.
- queue item이 단순 `in_progress`에서 `waiting_approval`, `stalled`, `resuming` 같은 상태를 표시해야 할 수 있다.

권장 적용 방식:

- 기존 SSM을 제거하지 말고, 먼저 `RunStep`을 shadow ledger로 추가한다.
- 첫 phase에서는 write-only로 기록하고, 동작 제어는 기존 orchestrator가 유지한다.
- shadow ledger와 기존 state가 불일치하는 케이스를 테스트로 잡은 뒤 driver 권한을 점진적으로 넘긴다.

### B. `AgentRuntimeNextStep` union

반영 내용:

- model output parser, SSM, orchestrator가 “다음 동작”을 문자열/상태 전이로 암묵 처리하지 않고 명시적 union으로 반환한다.

예시:

```ts
type AgentRuntimeNextStep =
  | { kind: 'continue_turn'; speakerId: string }
  | { kind: 'request_approval'; approvalKind: ApprovalKind; payload: unknown }
  | { kind: 'run_tool'; toolName: string; args: unknown }
  | { kind: 'handoff'; from: string; to: string; reason: string }
  | { kind: 'post_minutes'; snapshotId: string }
  | { kind: 'finish_meeting'; outcome: 'accepted' | 'rejected' | 'aborted' }
  | { kind: 'wait'; reason: string };
```

장점:

- `SYNTHESIZING`, `VOTING`, `EXECUTING`, `REVIEWING` 같은 현재 wait에 가까운 상태를 실제 driver로 만들기 쉽다.
- model parser가 엉뚱한 텍스트를 냈을 때도 “unknown next step”으로 안전하게 멈출 수 있다.
- provider별 output format 차이를 runtime adapter에서 흡수할 수 있다.

단점:

- `MeetingTurnExecutor`, parser, SSM transition, ApprovalService, ExecutionService 사이 interface가 크게 바뀐다.
- 모든 상태 전이 테스트를 다시 써야 한다.
- union이 너무 커지면 중앙 runtime이 비대해질 수 있다.

사이드 이펙트:

- 지금은 일부 fallback이 자동으로 이어지는 흐름이 approval wait나 failed state로 바뀔 수 있다.
- 회의가 더 자주 멈춰 사용자 결정을 기다릴 수 있다.
- 초기에는 결과 생성 속도가 느려진 것처럼 보일 수 있다. 대신 실패가 더 명확해진다.

권장 적용 방식:

- 먼저 `MeetingOrchestrator` 내부 private type으로 시작한다.
- 외부 stream/API 타입으로 바로 공개하지 않는다.
- parser에서 나온 결과를 next-step으로 변환하는 adapter test를 촘촘히 둔다.

### C. Symphony식 queue runtime reconciliation

반영 내용:

- `queue_items.status`는 사용자 관점 상태로 유지하고, 실행 lease/claim/retry/stall은 별도 runtime table 또는 runtime columns로 분리한다.
- app boot, meeting finalization, approval decision, provider failure 때 reconcile pass를 수행한다.

장점:

- crash recovery가 단순 pending rollback이 아니라 “연결된 meeting 상태를 확인한 뒤 resume/fail/release”가 된다.
- 여러 queue item이 `in_progress`가 되는 DB-level gap을 보완할 수 있다.
- retry/backoff와 circuit breaker의 근거가 명확해진다.

단점:

- DB migration이 필요하다.
- active meeting unique index, queue item active singleton, onFinal hook 사이 race condition을 다시 설계해야 한다.
- renderer queue UI가 더 많은 상태를 이해해야 한다.

사이드 이펙트:

- 앱 시작 시 예전처럼 자동으로 pending으로 되돌아가지 않는 item이 생긴다. 사용자는 `resuming`, `stalled`, `needs_review` 같은 새 상태를 볼 수 있다.
- 잘못된 reconcile은 중복 회의 시작이나 완료된 작업 재실행을 만들 수 있다.
- queue runner가 더 엄격해지면 기존 수동 테스트 흐름이 바뀔 수 있다.

권장 적용 방식:

- `queue_runtime` 또는 `run_steps`에 lease owner, lease expires, attempt, last heartbeat를 둔다.
- 기존 `queue_items.status`는 후방 호환을 위해 유지한다.
- boot recovery는 dry-run report를 먼저 만들고, 실제 mutation은 테스트가 충분히 생긴 뒤 활성화한다.

### D. Semantic stream event layer

반영 내용:

- `meeting-turn-token` 같은 raw token event는 유지하되, 그 위에 `run.step.started`, `run.step.waiting_approval`, `run.tool.requested`, `run.handoff.created`, `run.step.retried`, `run.step.finished` 같은 의미 event를 추가한다.

장점:

- UI timeline, approval detail, audit report가 같은 event vocabulary를 쓴다.
- provider별 출력 방식이 달라도 사용자에게 같은 형태로 보여줄 수 있다.
- 테스트가 token 문자열보다 의미 event를 검증할 수 있어 안정적이다.

단점:

- StreamBridge validation schema가 커진다.
- renderer hook과 store가 event versioning을 처리해야 한다.
- raw token event와 semantic event가 중복되어 event volume이 늘어난다.

사이드 이펙트:

- 이벤트 순서에 민감한 UI가 흔들릴 수 있다.
- cooldown/drop 정책이 더 복잡해진다.
- 기존 `use-meeting-stream`의 live turn 제거 타이밍이 달라질 수 있다.

권장 적용 방식:

- v1 semantic event를 additive로만 추가한다.
- renderer는 처음에는 개발자 timeline이나 debug panel에서만 사용한다.
- 기존 사용자 UI 전환은 event 안정화 뒤에 진행한다.

### E. Capability manifest

반영 내용:

- project, channel, meeting, provider turn마다 “허용된 파일 root, consensus path, tool set, network permission, CLI permission mode, approval policy, autonomy mode”를 하나의 manifest로 만든다.
- `PermissionService`, `PermissionFlagBuilder`, `ExecutionService`, CLI permission adapter가 이 manifest를 입력으로 공유한다.

장점:

- 실행 경계가 더 일관된다.
- external project + auto 금지 같은 규칙을 provider CLI args, file access, tool permission에서 동시에 강제할 수 있다.
- 승인 상세 UI가 “이 승인이 어떤 capability를 열어주는지”를 명확히 보여줄 수 있다.

단점:

- 현재 optional `ensureAccess`와 provider별 hardcoded config를 대체해야 한다.
- manifest schema가 잘못 설계되면 모든 실행이 막히거나 반대로 과도하게 열릴 수 있다.
- 기존 tests가 provider spawn args와 approval payload snapshot을 많이 다시 잡아야 한다.

사이드 이펙트:

- 지금은 허용되던 cwd command가 거부될 수 있다.
- CLI provider permission flag가 달라질 수 있어 Codex/Claude/Gemini behavior가 바뀐다.
- external/imported project 사용자에게 권한 확인이 더 자주 나타날 수 있다.

권장 적용 방식:

- 첫 단계는 read-only manifest preview다. 실제 enforcement는 하지 않고 로그와 approval detail에만 표시한다.
- 그 다음 `ExecutionService.runCommand`와 `applyPatch`에 manifest hash를 audit로 기록한다.
- 마지막에 optional `ensureAccess`를 required로 좁힌다.

### F. alex-core-invariants식 no-silent-fallback / invariant catalog

반영 내용:

- 프로젝트 invariant를 catalog로 만들고, 코드상 silent fallback 후보를 테스트와 정적 스캔으로 감시한다.
- 보안/승인/파일/큐 경계에서는 fallback을 fail-closed로 바꾸고, UI convenience fallback은 diagnostics에 남긴다.

장점:

- 코드 품질이 빠르게 좋아진다. 특히 “일단 warn하고 계속 진행”하는 곳의 위험도를 분류할 수 있다.
- 리뷰 기준이 명확해진다.
- AI가 만든 코드에서 흔한 legacy fallback, duplicate runtime, hidden bypass를 잡기 쉽다.

단점:

- false positive가 많을 수 있다.
- 초기에는 개발 속도가 느려진다.
- 모든 fallback이 나쁜 것은 아니므로 분류 기준을 잘못 세우면 유연성이 줄어든다.

사이드 이펙트:

- `consensus approval create failed -> immediate accepted finish` 같은 흐름을 막으면, 일부 회의가 완료 대신 failed/paused로 남는다.
- `streamBridge`가 snapshot accessor 누락 시 조용히 return하던 흐름을 error로 바꾸면 boot/test 환경에서 새 wiring requirement가 생긴다.
- provider persistent 실패 후 per-turn fallback을 error로 바꾸면 사용자에게 더 많은 실패가 노출된다.

권장 적용 방식:

- 바로 hard gate로 두지 말고 report-only scan부터 시작한다.
- invariant는 “보안/승인/실행 경계”와 “UI 편의 fallback”을 나눠 등급화한다.
- 경계 invariant는 테스트로, 스타일 invariant는 lint/report로 관리한다.

### G. Provider session policy 정리

반영 내용:

- provider별 session continuity, fallback, history source, token usage sink를 명시한다.
- CLI persistent session 실패 시 per-turn fallback이 어떤 history를 보내는지 runtime ledger에 기록한다.

장점:

- 같은 회의에서 provider 답변 품질이 흔들리는 이유를 추적할 수 있다.
- token usage와 LLM cost audit의 누락 가능성이 줄어든다.
- “회의 지시문이 DM으로 새는 문제” 같은 context leakage를 정책으로 관리할 수 있다.

단점:

- provider별 adapter complexity가 올라간다.
- session reset이 늘면 비용과 latency가 증가한다.
- 반대로 session reuse를 늘리면 컨텍스트 오염 위험이 커진다.

사이드 이펙트:

- 답변 품질이 좋아질 수도, 일시적으로 나빠질 수도 있다. 특히 CLI provider는 session 정책 변화에 민감하다.
- 기존 테스트 fixture의 expected output이 바뀔 수 있다.
- cost audit 숫자가 기존과 달라질 수 있다.

권장 적용 방식:

- provider call마다 `historyPolicy`, `sessionPolicy`, `fallbackPolicy`, `usageSinkPolicy`를 audit에 기록한다.
- 실제 policy 변경은 provider별로 한 번에 하나씩 한다.

## 5. 품질 최우선 적용 로드맵

작업 시간과 난이도를 고려하지 않는다면 다음 순서가 가장 좋다.

### Phase 0. 기준선 고정

- 현재 동작을 문서화하고 테스트로 고정한다.
- 특히 다음 시나리오를 먼저 테스트한다.
  - consensus approval 생성 실패 시 현재 fallback behavior
  - process restart 후 pending consensus approval decision 처리
  - queue `in_progress` + `started_meeting_id`가 있는 상태에서 boot recovery
  - CLI persistent 실패 후 per-turn fallback history
  - `ExecutionService.runCommand` cwd guard와 `ensureAccess` optional path
  - patch create path의 junction/symlink parent edge case

### Phase 1. Invariant catalog와 no-silent-fallback report

- `docs/decisions/` 또는 별도 `docs/invariants/`에 경계 invariant를 정리한다.
- 첫 실행은 report-only로 둔다.
- silent fallback 후보를 severity로 분류한다.

추천 high-severity 후보:

- 승인 생성 실패가 accepted finish로 이어지는 흐름
- command cwd root guard가 default wiring에서 비활성일 수 있는 흐름
- external/auto 금지가 provider args, execution access, tool access에서 다르게 적용되는 흐름
- approval kind schema coverage가 느슨한 흐름

### Phase 2. RunStep shadow ledger

- `runs`, `run_steps` 또는 기존 `meetings.snapshot_json` 확장 중 하나를 선택한다.
- 첫 단계에서는 기존 동작을 바꾸지 않고 기록만 한다.
- 각 meeting turn, provider call, approval wait, queue claim, file apply를 step으로 남긴다.

### Phase 3. Next-step driver 도입

- `MeetingOrchestrator` 내부에서만 next-step union을 쓴다.
- `SYNTHESIZING`, `VOTING`, `EXECUTING`, `REVIEWING`을 실제 driver가 처리하게 만든다.
- model parser output을 바로 state transition시키지 않고 next-step validation을 거친다.

### Phase 4. Semantic stream event 추가

- 기존 raw token event를 유지한다.
- run step event를 additive로 내보낸다.
- renderer에는 먼저 debug timeline이나 approval detail 연결 정보로만 표시한다.

### Phase 5. Queue runtime reconciliation

- queue lease/heartbeat/retry/stall을 별도 runtime layer로 둔다.
- boot recovery는 meeting state와 queue state를 함께 보고 결정한다.
- active singleton은 DB constraint 또는 transaction policy로 강화한다.

### Phase 6. Capability manifest enforcement

- manifest preview와 audit 기록부터 시작한다.
- 그 다음 `ExecutionService`, `PermissionFlagBuilder`, CLI permission adapter가 같은 manifest를 사용한다.
- 마지막에 optional access path를 줄이고 fail-closed 정책을 강화한다.

## 6. 하지 않는 것이 좋은 것

다음은 결과물 품질을 위해 피하는 편이 좋다.

- `openai-agents-python`을 직접 런타임 dependency로 가져오기. Rolestra는 Electron/TypeScript 앱이고, Python SDK 구조를 직접 얹으면 경계가 흐려진다.
- Codex/experimental sandbox app server 코드를 그대로 붙이기. 아이디어는 좋지만 Rolestra의 `ExecutionService`와 `PermissionFlagBuilder`를 우회하면 안 된다.
- provider CLI fallback을 전부 제거하기. fallback은 사용자 경험 안정성에 기여한다. 다만 silent fallback이 아니라 관찰 가능한 fallback으로 바꾸는 것이 맞다.
- stream event를 한 번에 교체하기. 기존 UI와 tests가 raw token event에 의존하므로 semantic event는 병행 추가가 안전하다.
- invariant scanner를 처음부터 CI hard gate로 걸기. 초반 false positive가 많을 가능성이 높다.
- Agestra 기반 자동화에 의존하기. 이번 조사 범위에서는 배제했다.

## 7. 예상 테스트 전략

구현 phase로 넘어간다면 다음 테스트가 필요하다.

- Unit
  - `AgentRuntimeNextStep` reducer/parser validation
  - approval kind별 decision router
  - capability manifest generation and denial reason
  - queue reconcile decision table
  - provider fallback policy

- Integration
  - meeting DONE -> consensus approval -> app restart -> approve -> minutes post -> meeting finish
  - queue item start -> meeting spawned -> process restart -> reconcile
  - CLI permission request -> approval timeout/reject/conditional
  - ExecutionService patch apply rollback failure case
  - command cwd outside workspace denial

- E2E
  - 회의 결과 승인 인박스 detail에서 run step, impacted files, consensus context가 연결되어 보이는지
  - queue 모드에서 다음 작업 자동 시작, 중단, 재개, 실패 표시가 일관적인지
  - provider persistent session fallback 발생 시 사용자에게 과하지 않지만 추적 가능한 진단이 남는지

## 8. 최종 판단

참고 프로젝트의 아이디어는 Rolestra에 반영할 가치가 크다. 단, “새 기능 몇 개를 붙이는 방식”이 아니라 런타임 안정성, 승인 복구성, 권한 일관성, 의미 기반 stream을 강화하는 구조 개선으로 가져와야 한다.

가장 우선순위가 높은 축은 다음 네 가지다.

1. `RunState` / `RunStep` 영속 ledger
2. `AgentRuntimeNextStep` driver
3. capability manifest
4. no-silent-fallback invariant catalog

이 네 가지가 자리 잡으면 Rolestra는 장시간 자율 실행, 다중 provider 회의, 승인 기반 파일 반영, 큐 연속 작업에서 훨씬 예측 가능해진다. 결과물 품질 측면에서도 모델 출력 자체보다 “출력을 검증하고 다음 행동으로 바꾸는 runtime”이 강해지므로 안정적인 개선 효과를 기대할 수 있다.

다만 반영 범위가 넓다. 실제 착수 시에는 먼저 기준선 테스트와 shadow ledger를 깔고, 기존 behavior를 관찰 가능한 형태로 고정한 뒤 점진적으로 driver 권한을 옮기는 방식이 가장 안전하다.
