## 1. 후보별 영향도 매트릭스

각 후보 sub-section 형식: 변경 의도 / 영향 spec / 영향 ADR / 영향 코드 / 영향 migration / 영향 test / R12-C2 인터락 / 선행 의존.

### A. RunStep 영속 기록부

**변경 의도:** 회의·승인·외부 AI 호출·도구 사용·파일 변경을 *한 권의 공식 기록부* 에 한 줄씩 영속 기록. 메모리에만 떠있는 진행 상태를 없앰.

**시스템이 어떻게 바뀌나:**

지금은 회의 진행 상태가 직원 머릿속(앱 메모리)에 떠있어요. 노트북을 닫거나 앱이 꺼지면 *어디까지 했는지* 일부 정보가 사라져 다시 켤 때 모호해지는 순간이 있습니다. 회의에서 합의된 결재가 처리 중인데 앱이 죽으면 "이 결재가 진짜 처리됐는지 / 다시 해야 하는지" 가 뚜렷하지 않아요.

새 모델은 **회의 한 건당 공식 기록부 한 권**을 만듭니다. 발언이 한 줄, 결재 요청이 한 줄, 도구 사용이 한 줄, 파일 변경이 한 줄. 각 줄에는 "지금 진행 중 / 결재 기다림 / 끝남 / 실패함 / 멈춤" 상태가 박혀 있어요. 앱이 꺼져도 다음에 켜면 기록부를 펼쳐서 *마지막 줄부터 정확히 재개* 합니다.

평소 사용자가 보는 변화:
- 회의 도중 앱 재시작해도 진행 상태가 정확히 복구됨
- 승인 인박스에 "이 결재가 어떤 회의의 어떤 단계에서 발생했는지" 가 한눈에 보임
- "이 파일이 왜 바뀌었는가 / 어떤 결재가 어떤 실행을 열었는가" 추적 가능
- 회의 결과가 누락되는 종류의 사고가 구조적으로 차단됨

### B. NextStep 결정 카드

**변경 의도:** AI 직원이 발언을 마친 직후 "다음에 무엇을 할지" 를 *명시된 7장 카드 중 하나* 로 답하게 만듦. 모호한 흐름이 자동으로 진행되는 대신 알 수 없는 결과는 안전하게 멈춤.

**시스템이 어떻게 바뀌나:**

지금은 AI 직원의 답변이 들어오면 시스템이 그 답변 형식을 보고 *알아서 다음 동작을 추측* 합니다. "이건 합의된 결과 같으니 다음 직원으로 넘기자 / 이건 실행 요청 같으니 결재로 보내자" 같은 판단이 답변을 해석하는 중간에 묻혀 있어요. 가끔 모호한 답변이 들어오면 시스템이 *그래도 어떻게든 진행* 하는 fallback 이 작동합니다 — 이게 의도와 어긋난 결과를 만드는 원인이 되기도 해요.

새 모델은 직원에게 발언 직후 **명시 카드 한 장** 을 같이 제출하게 합니다. 카드는 7종:
- 「계속 발언할게요」
- 「결재 받아주세요」
- 「이 도구 좀 써도 될까요」
- 「제 일이 아니라 ○○님께 넘길게요」
- 「회의록 정리하겠습니다」
- 「회의 끝납니다 (수락 / 거절 / 중단)」
- 「잠시 대기」

답변과 카드가 분리되어 있어 시스템은 *카드대로* 다음 동작을 합니다. 카드가 이상하거나 비어있으면 시스템은 임의로 진행하지 않고 **안전하게 대기 상태로 멈춰** 사용자 확인을 받아요.

평소 사용자가 보는 변화:
- 회의가 더 자주 "사용자 확인 대기" 로 멈춤 — 대신 *왜 멈췄는지가 분명*함
- 모호한 결과가 그냥 진행되는 일이 줄어듦
- 회의 결과 품질이 안정됨 (모델이 엉뚱한 텍스트를 만들어도 시스템이 *해석을 추측* 하지 않음)
- 초기에는 진행 속도가 약간 느려진 것처럼 느껴질 수 있으나, 실패가 눈에 보이게 됨

### C. 큐 운영 시스템 (현장 배치판)

**변경 의도:** 자율 큐를 단순 줄세우기에서 *현장 배치판* 으로 격상. 사용자가 보는 상태 (대기 / 진행 / 완료 / 실패 / 취소) 와, 시스템이 운영하는 상태 (담당 직원 / 임대 시간 / 재시도 횟수 / 멈춤 감지) 를 분리.

**시스템이 어떻게 바뀌나:**

지금은 자율 큐가 *단순한 줄세우기* 입니다. "다음 작업 / 진행 중 / 끝남" 정도만 표시되고, 한 작업이 멈추거나 실패하면 그 사실이 사용자에게 잘 보이지 않을 수 있어요. 앱을 재시작하면 진행 중이던 항목이 그냥 *대기 상태로 되돌아가* 다시 시작됩니다 — 이미 절반 처리된 작업이 처음부터 다시 도는 일이 생길 수 있어요.

새 모델은 큐를 **현장 배치판** 으로 봅니다. 배치판에는 두 줄이 있어요:
- **위쪽 줄 (사용자가 보는 상태)** — 대기 / 진행 / 완료 / 실패 / 취소
- **아래쪽 줄 (운영 정보)** — 누가 맡고 있나 / 언제 시작했나 / 임대 시간 얼마 남았나 / 몇 번 재시도했나 / 멈춰있는 시간 얼마

앱 재시작이나 회의 종료 / 결재 결정 / 외부 AI 실패 같은 일이 생기면 시스템이 자동으로 **배치판 점검** 을 돌립니다. 점검 결과에 따라 항목이 *다시 시작 / 다시 시도 / 멈춤 표시 / 재검토 필요* 같은 새 상태로 분류됩니다.

평소 사용자가 보는 변화:
- 큐 항목이 단순 "진행 중" 이 아니라 "재개 중 / 멈춤 / 재검토 필요" 같이 더 정확한 상태로 보임
- 같은 작업이 두 번 동시에 돌아가는 종류의 사고가 차단됨
- 앱 재시작 후 절반 처리된 작업이 처음부터 다시 시작되는 대신, 멈춘 자리에서 이어짐
- "왜 이 작업이 멈췄는지 / 얼마나 기다렸는지" 가 사용자에게 보임

### D. 의미 단위 알림 (situation cards)

**변경 의도:** 채팅창에 흐르는 글자 단위 알림 위에 *상황 카드* 단위 알림을 같이 흐르게 함. 기존 알림은 그대로 두고 추가만 하는 방식.

**시스템이 어떻게 바뀌나:**

지금은 AI 직원이 발언할 때 **글자 한 글자씩** 채팅창에 흘러요. "지금 답변 중" 정도는 보이지만, 그 직원이 *내부적으로 무슨 일을 하고 있는지* 는 잘 안 보입니다. 도구를 쓰려고 하는 중인지, 결재를 기다리는 중인지, 다른 직원에게 인계하는 중인지가 글자 흐름 속에 묻혀요.

새 모델은 글자 알림은 그대로 두고, 그 옆에 **상황 카드** 를 같이 흘립니다:
- 「발언 시작」
- 「도구 요청 중」
- 「결재 기다리는 중」
- 「○○님에게 인계됨」
- 「실패해서 재시도 중」
- 「발언 끝」

이 카드들은 채팅창에 직접 노출될 수도 있고, 운영자용 진단 패널에만 보일 수도 있습니다 (도입 초기에는 진단 패널 위주).

평소 사용자가 보는 변화:
- "이 직원이 지금 뭐 하는 중인지" 가 채팅창 / 진단 패널에서 즉시 파악됨
- 회의 timeline 이 단순 발언 흐름이 아니라 *업무 흐름* 으로 보임
- 외부 AI (Claude / Codex / Gemini) 가 답변 형식이 서로 달라도 사용자에게는 같은 카드 모양으로 정리되어 표시됨
- 운영 진단 / QA 검증 자료가 같은 알림 흐름을 공유 → 추적이 쉬워짐

### E. 임무 카드 (capability manifest)

**변경 의도:** AI 직원이 일을 시작하기 전에 *임무 카드* 를 받게 만듦. 카드 한 장에 "어느 폴더 / 어느 도구 / 어느 모드 / 어느 외부 접속 / 어떤 결재 정책" 이 모두 적혀 있고, 시스템의 모든 검문소가 같은 카드를 본다.

**시스템이 어떻게 바뀌나:**

지금은 AI 직원이 일을 시작할 때 *여러 검문소* (파일 검문 / 명령 검문 / 외부 AI 검문 / 결재 검문) 를 통과합니다. 그런데 각 검문소가 **자기만의 규칙** 으로 판단해요. 가끔 "외부 폴더 + 자율 모드 금지" 같은 같은 규칙이 검문소마다 *조금 다르게* 적용되는 일이 있어, 한쪽에선 통과되고 다른 쪽에선 막히는 미묘한 어긋남이 생깁니다.

새 모델은 일을 시작할 때 직원에게 **임무 카드** 한 장을 발급합니다:
- 「당신이 만질 수 있는 폴더 = X」
- 「당신이 쓸 수 있는 도구 = Y, Z」
- 「당신이 쓸 수 있는 외부 AI 모드 = …」
- 「당신이 받을 결재 정책 = …」
- 「당신의 자율 권한 = …」

모든 검문소는 *같은 카드* 를 보고 결정하므로 어긋남이 없습니다. 결재 화면에도 "이 결재가 어떤 임무 카드를 여는지" 가 같이 표시되어, 사용자가 *지금 무엇을 허락하는 건지* 명확하게 알 수 있어요.

평소 사용자가 보는 변화:
- 같은 권한 규칙이 여러 검문소에서 다르게 작동하는 일 차단
- 결재 인박스에서 "이 결재가 풀어주는 권한 범위" 가 한 카드로 보임
- 외부 폴더 / 자율 모드 같은 위험한 조합이 *모든 검문소에서 일관되게* 차단됨
- 권한 변경이 검문소 한 곳만 고쳐서는 부족 → 카드 발급 규칙을 고치게 되어, 부분 허용 / 부분 차단 같은 위험한 절충이 사라짐

### F. 6 헌법 + 검사관 (Invariant catalog)

**변경 의도:** 회사 헌법 6 조항을 명시 카탈로그로 정리 + 코드에 헌법 위반 의심 지점을 주기적으로 찾아 보고하는 *검사관* 도입. 안전 경계는 실패해도 *조용히 우회하지 않고 명확히 멈춤* 으로 정책 변경.

**시스템이 어떻게 바뀌나:**

지금도 Rolestra 에는 7가지 절대 금지 규칙이 있어요 (CLAUDE.md). "직원이 회사 본관 (Node API) 에 직접 못 들어감 / 결재 없이 파일 못 만짐 / 셸 명령 직접 못 씀" 같은 규칙들이죠. 다만 이 규칙들이 *왜 그런가, 어디서 어긋나기 쉬운가* 를 정리한 헌법 단위는 없습니다.

또 일부 흐름에서 일이 잘못되면 시스템이 *조용히 다른 길로 우회* 합니다. 예를 들어 결재 생성에 실패하면 "그냥 자동 수락하고 회의 종료" 같은 우회 처리가 있어요. 사용자가 보기엔 회의가 정상 종료된 것 같지만, 실제로는 *결재 단계가 없었던* 회의가 됩니다.

새 모델은 두 가지를 도입합니다:

**1) 헌법 6 조항 카탈로그**
- ① 진실은 한 곳 (SSoT) — 같은 정보를 여러 곳에서 따로 관리하지 않음
- ② 책임 분리 (SoC / SRP) — 한 부서가 두 가지 책임을 안 짊어짐
- ③ 일관성 — 화면·DB·기록부가 서로 모순되지 않음
- ④ 원자성 — 변경은 *완전히 끝나거나 / 전부 되돌리거나*
- ⑤ 멱등성 — 같은 요청을 여러 번 해도 같은 결과
- ⑥ 조용한 우회 금지 — 안전 경계는 실패하면 *명확히 멈춤*, 조용히 다른 길로 안 감

**2) 검사관 (정기 점검 스크립트)**
- 코드를 둘러보며 헌법 위반 의심 지점을 *후보 목록* 으로 보고
- 처음에는 hard gate 가 아니라 *리뷰 보조*. 시간이 지나며 등급별로 정책 강화
- 진짜 안전 경계 위반 (결재 / 실행 / 파일) 은 *fail-closed* 정책으로 변경 — 즉 실패하면 조용히 우회하지 않고 사용자에게 분명히 보고

평소 사용자가 보는 변화:
- 결재가 실패하면 회의가 *자동 수락* 되는 대신 "결재 실패로 회의 멈춤" 로 보임
- 외부 AI 가 실패해서 다른 모드로 우회될 때 *우회한 사실이 진단 로그에 기록* 됨
- 코드 리뷰 / QA 시점에 헌법 위반 후보 목록이 보고서로 나옴
- AI 가 만든 코드에서 자주 등장하는 *조용한 우회 / 죽은 코드 / 가짜 데이터* 가 빠르게 탐지됨

### G. 외주 의뢰서 정형화 (Provider session policy)

**변경 의도:** 외부 AI (Claude / Codex / Gemini) 에게 일을 맡길 때 *의뢰서* 형식을 명시. "이전 대화 어디까지 보여줄지 / 같은 세션 유지할지 새로 시작할지 / 실패하면 어떻게 할지 / 비용은 어디 적을지" 가 의뢰서에 박혀 있고, 운영 기록부에도 같이 남음.

**시스템이 어떻게 바뀌나:**

지금은 외부 AI 에게 일을 맡길 때 시스템이 *내부적으로 알아서* 처리합니다. "이전 대화 어디까지 보여줘야 하나 / 세션 끊겼으면 어떻게 복구하나 / 비용은 어디 모으나" 같은 정책이 코드 곳곳에 흩어져 있어요. 그래서 같은 회의 안에서도 외부 AI 답변 품질이 들쭉날쭉할 수 있습니다 — 한 번은 이전 맥락을 보고, 다음 번엔 새로 시작했을 수 있어요.

또 외부 AI 와의 세션이 실패하면 시스템이 *알아서 다른 모드로 우회* 합니다. 우회 자체는 안정성에 도움 되지만, 우회 후 외부 AI 가 보는 맥락이 바뀝니다 — 사용자는 *왜 답변 품질이 갑자기 달라졌는지* 알기 어려워요.

새 모델은 외부 AI 호출 한 건마다 **의뢰서 4 항목** 을 명시합니다:
- 「이전 대화 어디까지 보여줄지」 (전체 / 요약 / 새로 시작)
- 「세션 정책」 (같은 세션 유지 / 매번 새로 / 실패 시 새로)
- 「실패 시 정책」 (멈춤 / 다른 모드로 우회 + 사용자 알림 / 다른 모드로 조용히 우회 — 마지막은 헌법 ⑥ 위반이라 X)
- 「비용 기록처」 (어느 audit log 에 적을지)

이 의뢰서는 운영 기록부 (A 의 RunStep) 에 같이 남아, *왜 이 답변이 나왔는지 / 왜 우회됐는지* 사용자가 추적할 수 있어요.

평소 사용자가 보는 변화:
- 같은 회의에서 외부 AI 답변 품질이 흔들리는 이유가 추적 가능
- 외부 AI 세션 실패 → 다른 모드로 우회 시 "우회됨" 표시가 진단에 남음
- LLM 비용 audit 화면에 "어떤 정책으로 호출됐는지" 같이 표시 → 비용 누락 사고 차단
- 회의 지시문이 DM 으로 새는 종류의 *맥락 누설* 문제를 정책으로 관리 가능

### H. 인계 진행률 surface (사용자 요청)

**변경 의도:** 부서 → 부서 인계 시점에 "프로젝트 전체 진행률 + 인계 패키지" 를 한 화면에 정리. R12-C 의 인계 워크플로우 위에 *진행 가시성* 추가. 받는 부서가 *즉시 시작 가능한 상태* 로 인계 받음.

**시스템이 어떻게 바뀌나:**

지금은 부서별 회의가 끝나면 결과가 다음 부서 채널로 *그냥 넘어갑니다*. R12-C 채널-역할 design 에 인계 워크플로우 자체는 정의되어 있지만 — *받는 부서가 시작하려면 어디까지 정리됐는지* 파악하기 위해 이전 회의록 / 결재 / 산출물을 *직접 뒤져야* 합니다. 인계 패키지의 *정리/완성도* 가 보장되지 않아요.

또 사용자가 "지금 프로젝트 전체가 어느 단계까지 와 있는지" 알려면 여러 채널을 *왔다갔다* 해야 합니다. 기획에서 디자인으로, 디자인에서 구현으로 넘어갔는지 / 어느 단계가 정체됐는지 한눈에 보이지 않아요.

새 모델은 두 가지를 도입:

**1) 인계 패키지 정리 단계 (인계 *전*)**

기획 부서가 구현 부서로 넘기기 전에 한 단계가 추가됩니다 — "**인계 정리**":
- 결정문 (회의록의 [합의] / [제외]) 추출
- 산출물 첨부 (spec markdown / 와이어프레임 / 진행 사진)
- 받는 부서가 알아야 할 컨텍스트 요약
- 다음 부서가 *즉시 시작 가능한 상태* 인지 검증 (B 의 「회의록 정리」 카드와 결합)

이 정리가 끝나야 인계 카드가 "발송 가능" 상태가 됩니다. 정리되지 않은 미완성 인계는 *시스템이 막음* (헌법 ⑥ 조용한 우회 금지 적용).

**2) 프로젝트 전체 진행률 화면**

대시보드 또는 별도 surface 에 *프로젝트 전체 흐름* 이 보임:

```
[기획] ████████████░░░░░░░░  60% (회의 3/5 완료, 합의 도달, 정리 중)
   │
   ▼ 인계 대기
[디자인] ░░░░░░░░░░░░░░░░░░░░  0% (대기 중)
   │
   ▼
[구현] ░░░░░░░░░░░░░░░░░░░░  0%
   │
   ▼
[검토] ░░░░░░░░░░░░░░░░░░░░  0%
```

각 부서 단계마다 진행률 + 현재 상태 (회의 중 / 인계 대기 / 정체 / 완료) + 마지막 활동 시점이 표시됩니다.

평소 사용자가 보는 변화:
- "지금 프로젝트 어디까지 왔지?" 가 한 화면에서 즉시 파악
- 받는 부서가 *어떻게 시작해야 할지* 모르는 일이 사라짐 — 인계 패키지에 정리된 컨텍스트 + 산출물 + 결정문이 같이 옴
- 미정리된 인계가 *시스템에 의해 막혀* 받는 부서가 빈 손으로 시작하는 일 차단
- 어느 부서가 정체됐는지 / 왜 정체됐는지 (대기 / 사용자 결정 / 외부 AI 실패) 가 즉시 보임

## 3. 의존 그래프 (선행 / 후행 관계)

```
                    F (Invariant catalog)
                    │
                    │  *모든 후보의 평가 기준*
                    ▼
        ┌──────────────────────────┐
        │                          │
        │   A (RunStep ledger)     │◄─── B (NextStep union)
        │   ★ Foundational         │     (next-step 이 step 으로 영속됨)
        │                          │
        └────────────┬─────────────┘
                     │
        ┌────────────┼─────────────┬──────────────┐
        ▼            ▼             ▼              ▼
   D (Semantic    C (Queue        E (Capability   H (인계 진행률
    stream)        runtime)        manifest)       surface)
        │            │             │              │
        │            │             │              │
        └────────────┴─────────────┴──────────────┘
                     │
                     ▼
              G (Provider session policy)
              (audit sink — A/D/E/H 의 step/event/manifest/progress 기록 활용)
```

- **F** 는 횡단 평가 기준. 다른 후보 진행 시 항상 fail-closed / observable / SSoT 관점 적용.
- **A (RunStep ledger)** 는 B/C/D/E/H 모두의 시너지 기반. A 없이 B~H 도입 가능하지만 효과 반감.
- **B (NextStep union)** 는 A 와 짝. A + B 함께 도입 시 회의 lifecycle 재정의 완성. **B 의 「회의록 정리」 카드 → H 의 인계 정리 trigger**.
- **C/D/E** 는 A 위에서 자기 영역 강화. 서로 독립.
- **H** 는 A (진행률 데이터 소스) + B (인계 정리 trigger) 의존. 받는 부서가 알아야 할 컨텍스트가 RunStep 기록부에서 자동 추출됨.
- **G** 는 audit sink — 다른 후보의 step/event/manifest/progress 가 audit 에 들어가야 의미.
