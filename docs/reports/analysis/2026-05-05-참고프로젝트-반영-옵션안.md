## 1. R12-C2 sub-task 재구성 방식 (§7.1)

8후보를 R12-C2 의 미land sub-task 에 어떻게 흡수할지 결정.

### 옵션 R1 — 전체 재분해

미land 영역 (T10c + T11~T35) 을 모두 폐기 후 8후보를 *반영 단계 단위* (기반 → 시스템 → 화면 → 인계) 로 sub-task 새로 분해.

- **코드 품질**: 가장 깨끗. A (RunStep ledger) 와 F (검사관) 같은 기반 후보가 다른 후보의 *전제 조건* 으로 자연스럽게 깔린다. plan 1 권 = 단일 권위.
- **코드 정리**: 8후보 1개 = sub-task 1~3개로 *국소화* (후보가 여러 sub-task 에 흩어지지 않음).
- **동작 안정성**: T0~T10b land 자산 (옛 SSM 폐기 / OpinionService / MeetingMinutesService) 은 그대로 보존되므로 *이미 안정된 기반* 위에서 새 plan 시작.
- **위험**: spec (`channel-roles-design.md` §11.13~§11.18 회의 흐름 7 단계 + JSON schema 4 종) 이 새 plan 의 *입력* 으로 재정리되어야 한다. spec 자체는 살리되, plan 안에서의 *구현 순서* 가 달라짐.

### 옵션 R2 — 기반 신설 + 기존 흐름 조정

T11 진입 *전* 에 A + F 두 sub-task 를 신설. T11~T35 의 기존 부서별 회의 흐름 sub-task 들은 보존하고, 각 sub-task 안에 8후보 (B/C/D/E/G/H) 의 흡수 항목이 *추가 acceptance* 로 들어감.

- **코드 품질**: 큰 흐름이 두 갈래 — *부서 흐름* (기존) / *후보 흡수* (신규) — 가 sub-task 안에서 섞인다. sub-task 별 acceptance 라벨이 무거워져 의미가 흐려짐.
- **코드 정리**: 기존 plan 보존이 장점이자 단점. 같은 후보가 여러 부서 sub-task 에 분산되는 경우가 발생 (예: B 의 next-step union 은 *모든* 부서 회의 surface 를 갈아끼움).
- **동작 안정성**: A/F 가 먼저 깔리므로 다른 후보가 그 기반 위에서 동작. R1 과 동등.

### 옵션 R3 — additive plan 병행

T11~T35 그대로 진행. 8후보를 *별도 plan* 으로 묶어 R12-C2 와 병행 land.

- **코드 품질**: R12-C2 P2~P8 의 핵심 변경 surface (회의 오케스트레이터 / 턴 매니저 / SSM driver / 결재 경계) 와 후보 A/B/F 가 *동일 파일* 을 동시에 손댄다. 두 plan 이 교차하면서 sub-task 의 acceptance 가 서로의 결과를 가정 → 권위 분산.
- **코드 정리**: plan 두 개 = sub-task list 두 개 = land 우선순위 결정이 매번 필요. 의존 그래프가 plan 경계를 넘는다.
- **동작 안정성**: 두 plan 의 sub-task 가 같은 회의 surface 를 *서로 모르는 상태* 로 land 하면 회의 한 라운드 안에서 두 변경이 부분 적용된 상태 (예: A 의 ledger 영속만 진행되고 B 의 next-step 분기가 아직 land 안 됨) 가 발생 — *동작 안정성 회귀* 의 직접 원인.

→ R3 은 동작 안정성 회귀가 직접 발생하는 옵션이므로 옵션에서 *제외*.

### 추천 — R1

R2 도 가능하나, 평가 기준 (코드 품질 / 정리 / 동작 안정성) 모두에서 R1 이 우위:
- R2 는 큰 흐름 두 갈래 (부서 / 후보) 가 sub-task 안에서 섞이고, 같은 후보가 여러 sub-task 에 분산.
- R1 은 후보 1개 = sub-task 1~3개로 *국소화*. 후보별 acceptance 가 sub-task 경계와 정렬.
- T0~T10b land 자산은 양 옵션 모두 보존. R1 의 폐기 비용은 T10c + T11~T35 (모두 미land) 한정.

**추천: R1 (전체 재분해)**

## 2. B (NextStep) 자율 정책 (§7.2)

B 의 7장 카드는 안전 등급이 다르다:
- **안전군** — 「계속 발언」「대기」 — 결과가 회의 안에서 끝남
- **결과 전파군** — 「결재」「도구」「인계」「회의록 정리」「회의 종료」 — 회의 *밖* 에 결과를 만든다 (파일 변경 / approval 결정 / 다른 부서 진입 / 회의록 fix / 큐 진행)

### 옵션 B1 — 안전 카드만 자동

「계속 발언」「대기」 = 자동. 결과 전파 5 카드 = *항상 사용자 확인*.

- **코드 품질**: NextStep 분류가 *안전군 / 결과 전파군* 두 분류만 가짐 — 분류 규칙이 단순. 모호 카드 / 빈 카드 = 안전 wait 와 일관.
- **코드 정리**: 카드별 분기가 NextStep 분류 함수 한 곳에 집중. SSM driver 가 동일 분류 규칙 사용.
- **동작 안정성**: *최고*. 결과 전파 5 카드 = 회의 *밖* 결과 → 사용자 검토 필수. fall-through fallback 자체가 발생 안 함. AutonomyGate / ApprovalService 의 기존 검문소 정책과 일관.

### 옵션 B2 — 자율 모드 따름

자율 모드 (manual / auto_toggle / queue) 가 자동이면 7 카드 전부 자동. 수동이면 전부 확인.

- **코드 품질**: NextStep 분류 X. 단순. 단, 자율 모드가 "회의 흐름" 과 "결과 전파" 를 *동일 게이트* 로 묶음 — 두 영역의 안전 등급 차이가 정책에서 사라짐.
- **코드 정리**: 자율 모드 토글 1 개로 7 카드 전부 통제. 단순하나 *섞이는 통제*.
- **동작 안정성**: 자율 모드 = 자동 시점에 결재 / 인계 카드도 사용자 확인 없이 진행. ApprovalService 의 별도 게이트가 *부분적으로 우회됨*.

→ 결재 / 인계 카드의 자동 진행 = *절대 위반 금지 규칙 #3 (승인 없는 파일 반영 금지)* 와 충돌. 옵션에서 *제외*.

### 옵션 B3 — 카드별 사용자 정책

사용자가 7 카드 각각의 자동/확인을 토글.

- **코드 품질**: 7 카드 × 자동/확인 = 14 상태. 사용자 설정이 결재 / 인계 카드까지 자동으로 풀 수 있음 — B2 와 동일 충돌.
- **코드 정리**: 카드별 토글 UI + persist + IPC 까지 7 카드 전부 추가. 분기가 카드 수만큼 늘어남.
- **동작 안정성**: 사용자 실수로 결재 / 인계 = 자동 설정 가능 → ApprovalService 우회.

→ B2 와 동일 사유로 옵션에서 *제외*.

### 추천 — B1

B2 / B3 는 결재 / 인계 카드를 사용자 확인 없이 자동 진행할 가능성이 있어 절대 위반 금지 규칙 #3 와 충돌. 평가 기준 정책상 옵션에서 제외.

**추천: B1 (안전 카드만 자동)**

### B1 와 기존 정책의 인터락

B1 의 7 카드 분류는 *발언 직후 한 턴 동안의 다음 동작* 만 통제. 기존의 부서별 / 회의별 / 앱 전역 정책은 다른 레이어이므로 *공존 가능*. 사라지는 정책은 없다.

| 기존 정책 | surface | B1 와 관계 |
|---------|--------|-----------|
| **handoff_mode = check** (디폴트) | 부서 인계 gate | **정합** — B1 「인계」카드 = 사용자 확인 의미와 동일 |
| **handoff_mode = auto** (부서별 opt-in) | 부서 인계 gate | **계층** — B1 디폴트 = 사용자 확인. 사용자가 *명시적으로 켠 부서만* 인계 자동 우회. `channel-roles-design.md` §200 의 `DEFAULT 'check'` 와 정합 |
| **maxRounds (회의 cap, 5)** | 회의 라이프사이클 cap | **독립** — B1 「계속 발언」카드가 자동이어도 라운드 카운터 cap 도달 시 시스템이 「회의 종료」카드 강제 발행 또는 종료 트리거 |
| **자율 모드 (manual / auto_toggle / queue)** | 회의 시작·큐 진행 gate | **독립** — B1 = 회의 *내부* 발언 후 분류. 자율 모드 = 회의 *외부* 시작·큐 분류. 두 surface 분리 |

해석:
- handoff_mode auto 는 의미가 *살짝 좁아진다* — "B1 디폴트 위에서 부서별로 명시 켠 인계 자동 우회". 정책 자체는 유지, 의미는 더 명확해짐 (디폴트는 안전, 명시 opt-in 은 부서별).
- maxRounds / 자율 모드는 surface 가 다른 레이어 → 변경 없음.

**spec 갱신 시 명시할 점**: `channel-roles-design.md` §11.18 (회의 진행 단계) 안에 *카드 분류 → handoff_mode 우회 룰* 1 줄 명세 추가. 「인계」카드 도착 시 → handoff_mode = auto 부서면 자동 진행, 아니면 사용자 확인 모달.

## 3. H (진행률 surface) 위치 (§7.3)

H 는 두 surface 를 가진다:
- **합계 surface** — 프로젝트 전체 진행률
- **컨텍스트 surface** — 부서 간 인계 패키지 정리

### 옵션 H1 — 대시보드 통합

대시보드 메인에 *프로젝트 전체 진행률 패널* 추가. 인계 패키지는 인계 시점 모달 또는 인계 받는 채널 첫 진입 시 표시.

- **코드 품질**: 대시보드 = 사용자가 가장 많이 보는 surface. 진행률 = 프로젝트 단위 합계 정보 → 대시보드와 자연 정합. 대시보드 패널 추가는 R12-C 안에서 InsightStrip 같은 형태로 이미 패턴화.
- **코드 정리**: 진행률 패널 1개 + 인계 모달 1개 = surface 2개로 한정. 부서 채널 헤더에는 추가 변경 없음.
- **동작 안정성**: 진행률 = 합계 정보 → 채널 진입 / 인계 등 다른 흐름과 독립. 대시보드 진입 시 한 번 산출 / 부서 단위 변경 시 streaming update.

### 옵션 H2 — 인계 화면 별도

인계 surface 만 별도 화면. 받는 부서 채널 진입 시 *첫 화면* 으로 인계 패키지 표시.

- **코드 품질**: 인계 패키지 = 받는 부서 컨텍스트와 직접 연결되므로 *진입 시 첫 surface* 의 의미가 강함. B 의 결과 전파 「인계」 카드 → 사용자 확인 → 받는 부서 진입 → 첫 화면 패키지 흐름이 *명시적 게이트* 로 작동.
- **코드 정리**: 인계 화면 1개. 단, 프로젝트 전체 진행률을 어디 둘지 별도 결정 필요 — 단독으로는 합계 surface 가 미해결.
- **동작 안정성**: 인계 흐름의 명시적 게이트는 강하나, 합계 surface 가 없으면 사용자가 "전체가 어디까지 왔는지" 를 *조각조각 모아야* 함.

### 옵션 H3 — 부서별 채널 헤더 분산형

각 부서 채널 헤더에 *부서별 진행률 + 받은 인계 패키지 요약*.

- **코드 품질**: 6 부서 + 일반 채널 + DM 까지 헤더가 분산. 진행률 산출이 부서별 단위가 되어 *프로젝트 전체* 합계는 별도 계산 필요 → 결국 어딘가에 합계 surface 가 있어야 함.
- **코드 정리**: 6 부서 헤더 컴포넌트 모두 수정. 헤더 영역이 좁아 진행률 + 인계 요약을 함께 담기 어렵고, 인계 모달 / 패키지 상세 surface 가 별도로 필요.
- **동작 안정성**: 부서별 분산 = 각 surface 가 자기 부서 데이터만 의존 → 안정성 OK. 다만 *프로젝트 전체* 진행률을 보려면 부서별 헤더를 차례로 봐야 함.

### 추천 — H1 + H2 결합

- H1 단독: 인계 시점의 *받는 부서 진입 첫 화면* 이라는 강한 게이트 surface 가 사라진다 (모달로 약화).
- H2 단독: 합계 surface 미해결.
- H3 단독: 6 부서 분산 → 합계 surface 미해결 + 헤더 산포.

H1 (합계) + H2 (컨텍스트) 는 *서로 다른 정보* 를 다루므로 중복이 아니다. 두 surface 는 사용자의 두 질문 ("전체가 어디까지 왔나" / "이 부서가 무슨 패키지를 받았나") 에 1:1 대응.

**추천: H1 + H2 (대시보드 = 합계 / 받는 부서 채널 첫 화면 = 인계)**

## 4. F (검사관) 강도 (§7.4)

F = 6 헌법 (SSoT / SoC / Consistency / Atomicity / Idempotency / NoSilentFallback) 의 코드 표현. 검사 결과 처리 방식 결정.

### 옵션 F1 — report-only

위반 후보를 *보고서만* 생성. 빌드 / CI 영향 없음.

- **코드 품질**: 검사관이 *제안* 만 한다. 6 헌법 위반이 코드에 남아있어도 빌드 통과 → 헌법 보장이 *개발자 의지* 에 의존.
- **코드 정리**: 보고서 생성 surface 만 추가. 가장 가벼움.
- **동작 안정성**: 위반이 production 에 land 가능 → 헌법의 *동작 안정성 보장 부분* (NoSilentFallback / Atomicity / Idempotency) 이 강제되지 않음.

→ 절대 위반 금지 규칙 (특히 #2 silent fallback / mock 데이터) 의 코드 강제가 없어 *CLAUDE.md 절대 규칙* 과 *코드 검사* 사이에 갭이 발생.

### 옵션 F2 — 단계적 fail-closed

첫 phase = report-only (false positive 정리). 정리 완료 후 *안전 경계* (결재 / 실행 / 파일 / 마이그레이션) 만 fail-closed. UI 편의 fallback 같은 *비안전* 영역은 계속 report-only 유지.

- **코드 품질**: 안전 경계 위반 = 빌드 실패 → NoSilentFallback / Atomicity 가 *불가역 강제*. 비안전 영역은 점진 fail-closed 로 옮길 수 있는 옵션 보존.
- **코드 정리**: 검사관 룰이 *안전 경계 / 비안전* 두 분류만 가짐. 분류 규칙 = 절대 위반 금지 규칙 7 항목과 *1:1 정렬* 가능.
- **동작 안정성**: 안전 경계의 동작 안정성이 코드 표현으로 보장. false positive 정리 단계 (report-only first) 가 *룰 정확도* 를 먼저 검증한 후 강제로 전환되므로 검사관 자체가 빌드를 부당하게 막는 일이 줄어듦.

### 옵션 F3 — 즉시 hard gate

처음부터 모든 위반을 빌드 실패로.

- **코드 품질**: 헌법이 빌드에 *완전 강제*. 가장 엄격.
- **코드 정리**: false positive 정리 부담 + 검사관 룰의 정확도가 처음부터 100% 여야 함. 정확도가 부족하면 회피 패턴 (`// disable` 같은 회피 주석) 이 6 헌법에도 등장 → 헌법 자체가 *무력화*.
- **동작 안정성**: 검사관 자체가 false positive 로 빌드를 막으면 *그것 자체가 동작 불안정* (개발 흐름 차단).

### 추천 — F2

- F1: 절대 규칙 #2 (silent fallback / mock 데이터) 가 검사관에서 강제되지 않음 → CLAUDE.md 와 코드 표현 사이 갭.
- F3: false positive 가 빌드 차단 → 회피 주석 패턴 → 헌법 무력화. 동작 안정성 측면에서 *역효과*.
- F2: 안전 경계 = 절대 위반 금지 규칙 7 항목과 1:1 정렬. 비안전 영역은 점진 강화 가능. 헌법의 *동작 안정성 보장 부분만* 강제하는 *최소 강제 범위*.
