>
> **작성일**: 2026-02-14
> **기준 문서**: `설계-문서.md`, `implementation-plan.md`

## 1. 현황 분석

### 1-1. 아키텍처 분리도 평가

현재 코드베이스의 Electron 결합도를 분석한 결과:

| 계층 | Electron 의존 | 결합 지점 | 웹 전환 난이도 |
|------|:---:|---|:---:|
| 비즈니스 로직 (엔진/Provider/합의) | 없음 | — | 없음 |
| 데이터베이스 | `app.getPath()` 경로만 | `connection.ts` 1곳 | 사소 |
| 설정/비밀 | `app.getPath()` + `safeStorage` | `config-service.ts`, `secret-store.ts` | 보통 |
| 스트리밍 푸시 | `WebContents.send()` | `orchestrator.ts`의 `emit()` 1곳 | 쉬움 |
| IPC 라우터 | `ipcMain.handle()` | `router.ts` 1곳 | 보통 |
| UI (React) | `contextBridge` 경유 | `preload/index.ts` | 교체 |

**결론: 100개 이상의 소스 파일 중 Electron import가 있는 파일은 8개. 비즈니스 로직의 90%가 이미 전송 계층과 분리되어 있다.**

### 1-2. 재사용 가능한 기존 자산

- **`IpcChannelMap` (45+ 채널)**: 이미 요청/응답 계약이 타입 안전하게 정의됨 → HTTP 라우트로 1:1 매핑 가능
- **`StreamEventMap` (7개 이벤트)**: 푸시 이벤트 타입이 정의됨 → WebSocket/SSE 메시지로 그대로 사용
- **`RemoteServer` + `RemoteHandlers`**: Phase 5에서 구현한 읽기 전용 HTTP 서버 → 확장 기반
- **`RemotePermissionSet` / `RemoteAccessPolicy`**: 권한 모델이 이미 설계됨
- **React 프론트엔드**: 웹 브라우저에서 그대로 동작 (Electron 전용 API 미사용)

## 3. 핵심 설계 변경

### 3-1. StreamEmitter 추상화

현재 `ConversationOrchestrator`는 `WebContents`를 직접 받아 `webContents.send()`로 이벤트를 푸시한다.
이를 추상 인터페이스로 교체한다.

```typescript
// src/shared/stream-emitter.ts
import type { StreamEventName, StreamEventMap } from './stream-types';

/** Transport-agnostic event emitter interface. */
export interface StreamEmitter {
  emit<E extends StreamEventName>(event: E, data: StreamEventMap[E]): void;
  isAlive(): boolean;
}
```

**구현체**:

```typescript
// src/main/transport/ipc-stream-emitter.ts
import type { WebContents } from 'electron';

export class IpcStreamEmitter implements StreamEmitter {
  constructor(private webContents: WebContents) {}

  emit<E extends StreamEventName>(event: E, data: StreamEventMap[E]): void {
    if (!this.webContents.isDestroyed()) {
      this.webContents.send(event, data);
    }
  }

  isAlive(): boolean {
    return !this.webContents.isDestroyed();
  }
}

// src/main/transport/ws-stream-emitter.ts
import type { WebSocket } from 'ws';

export class WsStreamEmitter implements StreamEmitter {
  constructor(private ws: WebSocket) {}

  emit<E extends StreamEventName>(event: E, data: StreamEventMap[E]): void {
    if (this.ws.readyState === this.ws.OPEN) {
      this.ws.send(JSON.stringify({ event, data }));
    }
  }

  isAlive(): boolean {
    return this.ws.readyState === this.ws.OPEN;
  }
}
```

**Orchestrator 변경** (최소 수정):

```diff
- constructor(session: ConversationSession, webContents: WebContents) {
+ constructor(session: ConversationSession, emitter: StreamEmitter) {
    this.session = session;
-   this.webContents = webContents;
+   this.emitter = emitter;
  }

  private emit<E extends StreamEventName>(event: E, data: StreamEventMap[E]): void {
-   if (!this.webContents.isDestroyed()) {
-     this.webContents.send(event, data);
+   if (this.emitter.isAlive()) {
+     this.emitter.emit(event, data);
    }
  }
```

### 3-2. HTTP 라우터 (IpcChannelMap → REST 매핑)

`IpcChannelMap`의 45개 이상 채널을 HTTP 엔드포인트로 자동 매핑한다.

**매핑 규칙**:

```
IPC 채널                    HTTP 메서드    경로
─────────────────────────   ──────────    ─────────────────────────
app:ping                    GET           /api/app/ping
provider:list               GET           /api/provider/list
chat:send                   POST          /api/chat/send
chat:pause                  POST          /api/chat/pause
consensus:respond           POST          /api/consensus/respond
config:get-settings         GET           /api/config/settings
config:update-settings      PATCH         /api/config/settings
memory:search               POST          /api/memory/search
...
```

**구현 전략**: 기존 `src/main/ipc/handlers/` 핸들러를 그대로 호출하는 thin HTTP 어댑터

```typescript
// src/main/transport/http-router.ts
import type { IpcChannel, IpcChannelMap } from '../../shared/ipc-types';

/** IPC 핸들러 함수 시그니처 (기존 핸들러와 동일). */
type HandlerFn<C extends IpcChannel> = (
  data: IpcChannelMap[C]['request'],
) => Promise<IpcChannelMap[C]['response']> | IpcChannelMap[C]['response'];

/** 채널 → HTTP 핸들러 등록부. */
interface HttpRoute {
  method: 'GET' | 'POST' | 'PATCH' | 'DELETE';
  path: string;
  channel: IpcChannel;
  handler: HandlerFn<IpcChannel>;
}
```

### 3-3. WebSocket 스트리밍 프로토콜

### 3-5. 프론트엔드 어댑터

Renderer의 `useIPC` 훅이 현재 `window.arena.xxx()`를 호출하는데,
웹 모드에서는 HTTP/WS를 호출하도록 교체한다.

```typescript
// src/renderer/hooks/useTransport.ts

/** 전송 계층 추상화. Electron IPC와 HTTP/WS 중 환경에 맞게 선택. */
interface TransportClient {
  invoke<C extends IpcChannel>(
    channel: C,
    data: IpcChannelMap[C]['request'],
  ): Promise<IpcChannelMap[C]['response']>;

  onStream<E extends StreamEventName>(
    event: E,
    callback: (data: StreamEventMap[E]) => void,
  ): () => void;  // unsubscribe 함수 반환
}

function createTransportClient(): TransportClient {
  // Electron 환경: window.arena 존재
  if (window.arena) {
    return new IpcTransportClient();
  }
  // 웹 환경: HTTP + WebSocket
  return new HttpWsTransportClient(apiBaseUrl, wsUrl);
}
```

**IpcTransportClient** — 기존 preload 경유 (변경 없음)
**HttpWsTransportClient** — HTTP fetch + WebSocket

```typescript
class HttpWsTransportClient implements TransportClient {
  async invoke<C extends IpcChannel>(
    channel: C,
    data: IpcChannelMap[C]['request'],
  ): Promise<IpcChannelMap[C]['response']> {
    const { method, path } = channelToRoute(channel);
    const res = await fetch(`${this.baseUrl}${path}`, {
      method,
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${this.token}`,
      },
      body: method !== 'GET' ? JSON.stringify(data) : undefined,
    });
    if (!res.ok) throw new Error(await res.text());
    return res.json();
  }

  onStream<E extends StreamEventName>(
    event: E,
    callback: (data: StreamEventMap[E]) => void,
  ): () => void {
    // WebSocket 메시지 필터링
    const listener = (msg: MessageEvent) => {
      const parsed = JSON.parse(msg.data);
      if (parsed.event === event) {
        callback(parsed.data);
      }
    };
    this.ws.addEventListener('message', listener);
    return () => this.ws.removeEventListener('message', listener);
  }
}
```

## 4. 설정 계층 변경

### 4-1. Secret 저장소 분기

Electron의 `safeStorage` API는 웹 서버 모드에서 사용 불가.
환경에 따라 Secret 저장소를 분기한다.

```
Electron 모드  → safeStorage (OS 키체인)     [기존]
웹 서버 모드   → 환경변수 또는 암호화 파일     [신규]
```

```typescript
// src/main/config/secret-store.ts 수정
interface SecretStorageAdapter {
  encrypt(value: string): Buffer;
  decrypt(encrypted: Buffer): string;
  isAvailable(): boolean;
}

class SafeStorageAdapter implements SecretStorageAdapter { /* 기존 */ }
class EnvFileAdapter implements SecretStorageAdapter { /* 신규: .env.encrypted */ }
```

### 4-2. 데이터 경로

```
Electron 모드  → app.getPath('userData')     [기존]
웹 서버 모드   → CLI 인자 또는 환경변수로 지정  [신규]
```

```bash
# 웹 서버 모드 실행 예시
ARENA_DATA_DIR=/path/to/data \
ARENA_PORT=8443 \
ARENA_TOKEN=your-secret-token \
  node dist/server.js
```

## 8. 마이그레이션 전략

### 비파괴적 점진 전환

```
Step 1: StreamEmitter 추상화          ← 기존 코드 동작 불변
Step 2: HTTP/WS 서버 추가              ← 기존 Electron 앱에 영향 없음
Step 3: 프론트엔드 Transport 분기      ← 환경 감지로 자동 전환
Step 4: 보안/배포                      ← 서버 모드 전용
```

모든 단계에서 **Electron 앱은 기존대로 동작**한다.
웹 지원은 부가 기능으로 추가되며, 데스크탑 경험을 저해하지 않는다.

### 기존 Remote Access (Phase 5)와의 관계

Phase 5의 `RemoteServer`는 **읽기 전용** 원격 뷰어였다.
웹 지원은 이를 **완전한 양방향 웹 클라이언트**로 확장한다.

```
Phase 5 Remote (기존):  대화 조회 + 메모리 검색 (4개 엔드포인트)
Web Support (확장):     전체 IPC 채널 노출 (45+ 엔드포인트) + 실시간 스트리밍
```

기존 `RemoteServer`의 인증/세션/감사 인프라는 그대로 재사용한다.

## 9. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| WebSocket 연결 불안정 (모바일, 약한 네트워크) | 토큰 유실, 대화 끊김 | sequence 번호 기반 갭 감지, 자동 재연결, 메시지 버퍼 |
| 외부 노출 보안 취약점 | API 키 유출, 무단 접근 | HTTPS 필수, Rate limiting, IP 허용 목록, 토큰 만료 |
| SQLite 동시 접근 (Electron + 웹 동시 사용) | DB 잠금, 쓰기 충돌 | WAL 모드 활성화 (이미 적용), 쓰기 직렬화 |
| `safeStorage` 미사용 시 키 보안 저하 | API 키 평문 노출 | 암호화 파일 + 환경변수 조합, 파일 권한 600 |
| CLI Provider의 서브프로세스 관리 | 서버 모드에서 프로세스 라이프사이클 | 기존 Provider 라이프사이클 그대로 유지 (서버 프로세스 = Main Process) |
