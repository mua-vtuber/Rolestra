## 1. AI 참여자 관리

### 1-1. AI 유형

3가지 유형의 AI를 지원한다.

| 유형 | 예시 | 설정 방식 |
|------|------|----------|
| API | OpenAI, Anthropic, Google 등 | 드롭다운으로 추가, API 키 입력, X 버튼으로 삭제 |
| CLI | Gemini CLI, Claude Code, Codex CLI | "설치하고 인증한 뒤 사용해주세요" 안내 표시 |
| 로컬 LLM | Ollama 등 | "설치해주세요" 안내 표시 |

### 1-2. AI 설정

- 각 AI별 설정 모달 제공
  - **이름**: 사용자가 자유롭게 지정
  - **페르소나**: 시스템 프롬프트 (성격, 전문분야, 말투 등)
  - **사용 모델**: 해당 provider에서 사용할 모델 선택
  - **로컬 접근 권한**: 기본 권한 설정 (읽기/쓰기/실행)
- 로컬 LLM은 중복 생성 가능 (같은 Ollama라도 모델/페르소나를 다르게 하여 여러 개 등록)
- **Capability 선언**: Provider는 capability를 선언하며 (`streaming`, `resume`, `tools`, `web-search`, `multimodal` 등), UI/엔진 분기는 Provider 이름이 아닌 capability 기반으로 처리

### 1-3. 메인 화면 참여자 표시

- 설정에서 등록된 AI만 참여자 목록에 표시
- 참여자 선택/해제로 대화 참여 조절
- **빈 상태 처리 (중요)**:
  - 설정된 AI가 없으면 참여자 목록을 표시하지 않음
  - 대화 전송 버튼 비활성화
  - 대화 화면에 "AI가 없습니다. 설정해주세요." 메시지 표시

## 2. 대화 시스템

### 2-1. 기본 원칙

- 사용자도 AI 참여자와 **동등한 대화 참여자**로 취급한다
- 사용자를 특별 취급하지 않는다 (주인 구분 없음)
- 페르소나 기본 템플릿:

```
[한국어]
이 대화의 핵심 규칙:
- 다른 참여자에게 단순 동의하지 마라
- 빈 동의 표현을 사용하지 마라 ("좋은 지적이다", "동의합니다" 등)
- 진심으로 동의할 때만 동의를 표현하라
- 주장은 구체적 근거로 뒷받침하라
- 논리적 허점은 직접적으로 지적하라
- 자신의 입장의 약점에 대해 지적으로 정직하라

[English]
CRITICAL RULES FOR THIS CONVERSATION:
- Do NOT simply agree with the other participant
- Do NOT use empty agreement phrases ("Great point", "I agree", etc.)
- Only acknowledge agreement when you genuinely agree
- Support arguments with concrete evidence
- Point out logical gaps directly
- Be intellectually honest about your position's weaknesses
```

- **페르소나 비공유 원칙**: 다른 AI의 페르소나(시스템 프롬프트)는 전달하지 않음. AI의 태도와 성격은 대화에서 자연스럽게 드러남.

### 2-1-A. AI 간 구조화 통신

AI 간 메시지는 구조화된 JSON 형식으로 전달한다.

| 모드 | 필드 |
|------|------|
| 대화 모드 | `name`, `content`, `mode_judgment` (conversation/work), `judgment_reason` (출력 전용) |
| 작업 토론 | `name`, `opinion`, `reasoning`, `agreements` (다른 AI별 동의여부) |
| 리뷰 | `name`, `review_result` (approve/request_changes), `issues`, `comments` |

- AI 이름은 사용자 설정에서 지정한 표시명
- 다른 AI의 페르소나는 포함하지 않음 (페르소나 비공유 원칙)
- JSON 파싱 실패 시 원문을 `content`로 취급, `mode_judgment`은 `'conversation'` 기본값
- 파싱 재시도 횟수는 `parseRetryLimit` 설정으로 제한

### 2-2. 일반 대화

- 사용자와 AI들이 자유롭게 대화
- AI끼리는 비판적으로 토론
- 파일/코드 작업 없이 순수 대화

### 2-3. 작업 대화 모드

대화 중 AI들이 `mode_judgment: "work"` 판단을 과반수 내리면, 작업 모드 전환을 제안한다.
- 전환 제안 시 사용자에게 팝업 표시: [작업 모드로 전환] / [대화 계속]
- 사용자 승인 시 작업 토론 모드로 진입
- 사용자 거부 또는 타임아웃 시 대화 모드 유지
- 프로젝트 컨텍스트를 AI에게 제공하고, 대화 주제와 함께 전달할 수 있다

#### 프로젝트 컨텍스트 모드

참여자의 provider 능력에 따라 자동 분기:

**폴더 모드** (CLI 또는 도구지원 Local이 1명 이상):
- 프로젝트 **폴더** 선택 → AI가 자유롭게 파일 탐색
- API AI는 대화 흐름에서 다른 AI의 인용/분석을 통해 컨텍스트 획득

**파일 모드** (API만 / 도구 미지원 Local만):
- **개별 파일** 선택 → 프롬프트에 내용 직접 주입
- 알림: "선택한 파일만 AI에게 전달됩니다. CLI 도구를 추가하면 프로젝트 전체 탐색이 가능합니다."

참여자 변경 시 모드 자동 재판정 (CLI 추가 → 폴더 모드 승격, CLI 제거 → 파일 모드 제한).

#### 실행 방식 분기

**1:1 대화 (AI 1명만 참여)**
- AI가 직접 파일 수정 (승인 없이 즉시 실행)

**Arena 대화 (AI 2명 이상 참여) — 합의 기반 실행**

```
[대화 단계] CONVERSATION
  AI들이 JSON 구조화 대화, 매 라운드 mode_judgment 집계
      ↓ (과반수 "work")
[모드 전환 제안] MODE_TRANSITION_PENDING
  사용자에게 팝업: [작업 모드로 전환] / [대화 계속]
      ↓ (승인)
[작업 토론] WORK_DISCUSSING
  AI들이 의견/근거/동의여부를 JSON으로 교환
      ↓
[취합 단계] SYNTHESIZING
  취합자가 취합 문서 작성
  - aggregatorStrategy: strongest (기본) / last-speaker / designated / round-robin
      ↓
[투표 단계] VOTING
  각 AI에게 취합 문서 제시 + "동의하는가? 코멘트를 달라"
      ├── 전원 동의 → CONSENSUS_APPROVED
      ├── 반대 (재시도 남음) → WORK_DISCUSSING 복귀
      └── 반대 (재시도 소진) → FAILED
      ↓
[작업자 선택] CONSENSUS_APPROVED
  사용자가 작업을 수행할 AI 선택
      ↓
[실행 단계] EXECUTING
  선택된 AI에게 쓰기/실행 권한 부여, 작업 수행
  다른 AI는 읽기 전용 + "작업 금지" 시스템 프롬프트
      ↓
[리뷰 단계] REVIEWING
  작업자 권한 회수, 변경분(diff)을 다른 AI에게 전달하여 리뷰
      ↓
[사용자 판단] USER_DECISION
  리뷰 결과 + diff를 사용자에게 표시
  사용자: [수락] / [재작업] / [다른 AI 재할당] / [중단]
      ↓
[완료] DONE

※ 전체 흐름은 SessionStateMachine으로 구현
   - 12개 상태: CONVERSATION, MODE_TRANSITION_PENDING, WORK_DISCUSSING,
     SYNTHESIZING, VOTING, CONSENSUS_APPROVED, EXECUTING, REVIEWING,
     USER_DECISION, DONE, FAILED, PAUSED
   - 권한 변경은 상태 전이의 부수 효과로만 발생 (상태 전이 없이 권한 변경 불가)
   - 동의 실패 반복은 maxRetries로 제한, 각 단계는 타임아웃 적용
   - 합의 실패 시: AI별 대안/근거를 표 형태로 렌더링하여 사용자가 선택
```

#### 라운드 설정

**메인 화면에 배치** — 대화마다 다르게 조절 가능

```
┌──────────────────────────────────────────────────────┐
│ 라운드  [  5  ]  ☐ 합의될 때까지                     │
└──────────────────────────────────────────────────────┘

- 숫자 입력: 토론 라운드 수 지정
- "합의될 때까지" 체크 시:
  - 라운드 제한 없음
  - 숫자 입력 필드 회색으로 잠김 (비활성)
  - 합의가 도출되면 자동 종료
```

#### 사용자 참여 방식

- 사용자는 AI처럼 턴을 기다리지 않는다
- **언제든 자유롭게 대화에 끼어들 수 있음**
- AI 대화 사이에 2초 대기 시간 — 사용자가 끼어들 여유 제공
  - 대기 중 `...` 타이핑 애니메이션 표시 (다음 AI 준비 중 암시)
  - 사용자가 메시지를 보내면 대기 취소, 사용자 메시지 먼저 처리

#### 대화 표시 UX

- **스트리밍 표시**: 모든 AI 응답을 토큰 단위로 실시간 표시
  - API: SSE/스트리밍 응답 활용
  - CLI: stdout 실시간 파싱하여 스트리밍처럼 표시
  - 로컬 LLM: 기본 스트리밍 지원
- **"생각중..." 애니메이션**: AI에게 전달 후 ~ 첫 토큰 도착 전까지 표시
- **일시정지/재개 버튼**: Arena 대화 중 언제든 흐름을 멈추고 재개 가능

### 2-4. 추가 UX 기능

#### 대화 컨트롤
- **대화 분기(Fork)**: 특정 메시지 시점에서 "여기서 다시" — 다른 방향으로 토론 재시작
  - 메시지 계층 구조(`parentMessageId`, `branchId`, `branchRootMessageId`)로 분기 추적
  - 분기 전 메모리는 공유, 분기 후 독립
- **메시지 핀/북마크**: 중요한 발언/결정 사항 핀 고정
  - 핀 시 토픽 선택 (기술/결정/선호/맥락)
  - 메모리에 자동 저장
  - 이미 자동 추출된 기억에 핀하면 → importance_score 부스트

#### 정보 표시
- **AI별 응답 시간**: 각 메시지 옆에 소요 시간 표시 (느린 provider 파악)
- **토큰 사용량**: 대화별 누적 토큰 (API 비용 감각 유지)

#### 대화 내 검색
- **Ctrl+F 검색**: 현재 대화 내 키워드 검색

#### 작업 공간 구조

```
📁 사용자가 지정한 프로젝트 폴더
  └── 권한은 AI별 설정에 따름 (읽기/쓰기/실행)

📁 .arena/workspace/  (자동 생성, 숨김 폴더)
  ├── drafts/          AI들이 자유롭게 쓰는 공간
  ├── proposals/       취합 문서, 동의 코멘트
  └── approved/        승인된 최종본

  → .arena/workspace/는 전원 읽기/쓰기 자동 부여
  → 실제 프로젝트 폴더 반영은 승인 후에만
  → git 자동관리는 기본 OFF (설정에서 ON 가능)
```

#### 파일 권한 설정

**기본 권한 (대화/토론 모드)**:
- 모든 AI: 프로젝트 폴더 읽기 전용
- `.arena/workspace/`: 전원 읽기 + 쓰기 자동 부여

**작업 중 권한 (EXECUTING 상태)**:
- 작업자 AI: 프로젝트 폴더 읽기 + 쓰기 + 실행
- 다른 AI: 프로젝트 폴더 읽기 전용 유지

```
상태별 권한 매트릭스:
┌──────────────────┬──────────────┬──────────┬──────────────────┐
│ 상태             │ 작업자 AI    │ 다른 AI  │ .arena/workspace │
├──────────────────┼──────────────┼──────────┼──────────────────┤
│ CONVERSATION     │ 읽기         │ 읽기     │ 읽기 + 쓰기      │
│ WORK_DISCUSSING  │ 읽기         │ 읽기     │ 읽기 + 쓰기      │
│ CONSENSUS_APPROVED│ 읽기         │ 읽기     │ 읽기 + 쓰기      │
│ EXECUTING        │ 읽기+쓰기+실행 │ 읽기   │ 읽기 + 쓰기      │
│ REVIEWING        │ 읽기         │ 읽기     │ 읽기 + 쓰기      │
│ DONE             │ 읽기         │ 읽기     │ 읽기 + 쓰기      │
└──────────────────┴──────────────┴──────────┴──────────────────┘
```

- 권한 변경은 `SessionStateMachine` 상태 전이의 부수 효과로만 발생
- CLI별 권한 어댑터가 각 CLI 도구의 권한 제어 방식에 맞게 매핑
- **상태 전이 없이 권한 변경 불가** (안전 불변 조건)

#### 실행 안전 계층

**ExecutionService (API/Local AI)**:
  1. dry-run diff 생성
  2. 사용자 승인
  3. atomic apply
  4. 실패 시 rollback

**CLI 권한 어댑터 (CLI AI)**:
  - CLI 도구는 직접 파일 접근 가능 → ExecutionService 우회
  - CLI별 권한 어댑터로 제어:

| CLI | 읽기 전용 모드 | 작업자 모드 | 제어 방식 |
|-----|--------------|-----------|----------|
| Claude Code | `--allowedTools` 제한 | 제한 해제 | CLI 플래그 |
| Codex | `--approval-mode suggest` | `--approval-mode auto-edit` | CLI 플래그 |
| Gemini CLI | 시스템 프롬프트 | 시스템 프롬프트 | 프롬프트 전용 |
| Aider | `--no-auto-commits` | 플래그 해제 | CLI 플래그 |

  - 플래그 미지원 CLI: 시스템 프롬프트 + 사후 diff 검증으로 fallback
  - 작업 완료 후 diff 생성 → 리뷰 AI에게 전달 (전체 코드가 아닌 변경분만)

**공통 명령 실행 안전**:
  - 허용 명령 allowlist
  - 위험 패턴 차단 (blockedPatterns)
  - 실행 시간/출력 상한
  - 모든 실행 시도는 감사 로그 기록

## 4. 메모리 시스템

### 4-1. 설계 방향

- bara_system의 지식 그래프 메모리를 기반으로 하되, 더 적극적으로 저장/활용
- Bara는 소셜봇이라 토큰 절약 중심이었으나, Arena는 대화 품질이 핵심이므로 넉넉하게 사용
- **단계적 접근**:
  - Phase 3-a: FTS5 전문검색 + 핀 + Regex 자동 추출 (외부 의존 없음)
  - Phase 3-b: 임베딩 + 하이브리드 검색 + 반성 엔진 + 진화

### 4-2. 토픽별 분리 저장

```
메모리 토픽:
├── technical/     기술 문제 해결 기록
├── decisions/     설계 결정 히스토리
├── preferences/   사용자 선호/습관
└── context/       프로젝트별 맥락
```

### 4-3. 핵심 기능 (bara_system 참고)

- **하이브리드 검색**: 벡터 유사도 + FTS5 전문검색 + 지식그래프 관계 검색
- **스코어링**: Stanford 3-factor (최신성 × 관련성 × 중요도)
- **메모리 진화**: 유사 노드 병합, 오래된 기억 정리, 중요도 재계산
- **반성 엔진**: 누적 기억 분석 → 인사이트 생성
- **토큰 예산 컨텍스트 조립**: 프롬프트 토큰을 비율로 분배
- **토큰 카운터**: `js-tiktoken` 기반 정확 카운팅
- **운영 컬럼**: `embedding_version`, `extractor_version`, `source_hash`, `dedupe_key`, `deleted_at`

### 4-4. 메모리 저장 경로 (이중 경로)

```
경로 1: 자동 저장
  대화 진행 중 LLM이 자동으로 중요 사실/결정 추출
  ├── Regex 빠른 경로: "~로 가기로 했다", "~를 쓰자" 등 패턴 감지
  └── LLM 구조화 추출: 대화 맥락에서 사실/선호/관계 자동 추출

경로 2: 수동 저장 (핀)
  사용자가 메시지에 핀 → 토픽 선택 → 메모리 저장
  ├── 🔧 기술 (technical)
  ├── 📋 결정 (decisions)
  ├── 👤 선호 (preferences)
  └── 📁 맥락 (context)

연계:
  - 자동 추출된 기억에 사용자가 핀 → importance_score 부스트
  - 핀은 자동 저장의 보완이지 대체가 아님
```
