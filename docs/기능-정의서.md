# Rolestra v3 — 기능 정의서

> **목적:** Rolestra v3 가 사용자에게 제공하는 기능을 비즈니스 관점에서 정의.
> 기술 설계는 [`docs/설계-문서.md`](설계-문서.md), 단일 권위 spec 은 `docs/specs/2026-04-18-rolestra-design.md`.
>
> **재작성:** 2026-04-27 (R11 Task 13). v2 (AI Chat Arena) 잔재 0. ko-only.

---

## 1. Rolestra 메타포 — "AI 직원이 일하는 회사"

Rolestra 는 사용자가 **여러 AI 를 회사의 직원처럼** 운영하는 데스크톱 애플리케이션이다.

| 메타포 | 의미 |
|--------|------|
| 회사 | ArenaRoot — 사용자 home 의 단일 디렉토리 (`~/Rolestra/`) |
| 직원 | AI provider — Claude Code / Codex / Gemini CLI / Anthropic API / OpenAI API / Local Ollama 7 종 |
| 프로젝트 | 폴더 — 신규 / 외부 연결 / 가져오기 3 가지 방식 |
| 채널 | 메신저 그룹 — 시스템 채널 (`#회의록` / `#공지` / `#일반`) + 사용자 정의 + DM |
| 회의 | 직원 2 인 이상 모여 합의 도출 (ConsensusStateMachine 12 상태) |
| 결재함 | 회의 결과 + CLI 권한 + 모드 변경 + 실행 결과 + 회로 차단 5 종 승인 인박스 |
| 큐 (자율 작업) | 사용자가 AI 에 작업을 라인 단위로 위임 — 백그라운드 실행 |
| 출근 / 외근 / 점검 | 직원 상태 — `online` / `offline` / `connecting` / `field_work` / `inspection` |
| 회의록 | 회의 종료 시 자동 LLM 요약 + 메타 헤더 (참여자 / 주제 / SSM / 경과 / 투표) |

목표는 **사용자가 AI 한 명과 채팅하는 단방향 도구가 아니라, 여러 AI 를 동시에 가용하면서 합의 + 승인 + 자동 실행 + 비용 가시화 까지 통합한 워크플로 도구**.

## 2. 핵심 기능 — 11 phase 누적

### 2.1 직원 (AI Provider) 관리

7 가지 provider:

| 종류 | 예시 | 설정 |
|------|------|------|
| API | Anthropic Claude / OpenAI GPT / Google Gemini / OpenRouter | API 키 입력 (safeStorage 암호화 저장) + 모델 선택 |
| CLI | Claude Code / Codex / Gemini CLI | 설치 + 인증 후 자동 detect |
| Local | Ollama / llama.cpp / vLLM | localhost endpoint 설정 |

각 직원별 설정:
- **표시 이름** (`displayName`) — 사용자가 자유 지정
- **역할** (`role`) — 페르소나 (PM / 개발자 / 리뷰어 / 등)
- **개성** (`personality`) — 시스템 프롬프트 (말투 + 성격 + 전문 분야)
- **전문 분야** (`expertise`) — 직원 검색 / 회의 자동 초대 시 매칭 키워드
- **커스텀 아바타** — 5 MB 이하 png/jpg/jpeg/webp/gif (`<ArenaRoot>/avatars/<providerId>.<ext>`)
- **Capability 자동 인식** — `streaming` / `resume` / `tools` / `web-search` / `summarize` 5 종

직원 전체 보기 surface 3 종 (R8 D5 — 동일 Popover):
- 메시지 버블 아바타 클릭
- MemberRow 클릭 (MemberPanel)
- PeopleWidget 클릭 (Dashboard)

편집 시점은 **Popover (보기) → "편집" 버튼 → Modal (편집)** 2 단계 (R8 D6).

### 2.2 출근 상태

| 상태 | 의미 |
|------|------|
| `online` | 출근 — 회의 + 메시지 응답 가능 |
| `offline` | 퇴근 — 비활성 |
| `connecting` | 출근 중 — Warmup 진행 (5 초 timeout) |
| `field_work` | 외근 — 외부 시스템 호출 중 (lazy timeout 평가, R9 D7) |
| `inspection` | 점검 — provider 상태 이상 |

부팅 시 모든 provider 를 `Promise.allSettled` + 5 초 timeout 으로 fire-and-forget warmup (R8 D3) — 첫 화면 블락 방지.

회의 진행 중 `online` 이 아닌 멤버는 그 턴만 skip, 회의는 계속 (R8 D4).

### 2.3 프로젝트 관리

3 가지 생성 방식 (`+ 새 프로젝트` 모달, ProjectRail + Hero 빠른 액션 두 위치, R4 D3):

| 방식 | 설명 | 권한 모드 제약 |
|------|------|---------------|
| 신규 (`new`) | `<ArenaRoot>/projects/<slug>/` 새 폴더 생성 | `auto` / `hybrid` / `approval` 모두 가능 |
| 외부 연결 (`external`) | 사용자 외부 폴더를 Windows junction / macOS+Linux symlink 로 연결 | `external + auto` 비활성 (path-guard 안전 보장 어려움) |
| 가져오기 (`imported`) | 외부 폴더 자체를 ArenaRoot 안으로 복사 | `auto` / `hybrid` / `approval` 모두 가능 |

junction TOCTOU 검증 — 사용자가 junction 을 도중에 다른 폴더로 바꿔도 path-guard 가 realpath 비교로 차단 (CA-3, R4 D4).

활성 프로젝트 전환 — ProjectRail 클릭 + ShellTopBar subtitle 표시 + persist (`rolestra.activeProject.v1`).

### 2.4 채널 + 메신저

3 가지 channel kind:

| kind | 자동 생성? | 편집? |
|------|----------|-------|
| `system` (`#회의록` / `#공지` / `#일반`) | 프로젝트 생성 시 자동 (R5 D2) | rename / delete 비활성 |
| `user` (사용자 정의) | 수동 (`+`) | rename / delete 가능 |
| `dm` (1:1 DM) | DmCreateModal (`+`) | rename 비활성 / delete 가능 (R10 D1, channels.kind='dm' 재사용) |

메신저 surface (themeKey 3-way DOM 분기, R5 D3):
- ChannelRail (좌측 채널 리스트)
- Thread (메시지 목록 + DateSeparator + compact mode + live turn replace, R6 D6)
- ChannelHeader (채널 메타 + 회의 시작 버튼)
- Message + SystemMessage (themeKey 별 avatar/header/mono-prefix 분기)
- MeetingBanner (활성 회의 표시, retro 별도 JSX)
- ApprovalBlock (인라인 결재 — 회의 중 cli_permission 발생 시)
- Composer (메시지 입력 + 첨부 + Slash command)
- MemberPanel (우측 멤버 목록 + 출근 상태)

메시지 검색 (R10 Task 2) — Radix Dialog 모달 (D2). FTS5 + ShellTopBar 🔍 진입 + 결과 row → 채널 deep-link.

### 2.5 회의 (MeetingSession)

participants ≥ 2 보장 (R6 D7 — 1:1 DM 은 회의 X). ConsensusStateMachine 12 상태 구동.

#### 회의 흐름 (12 state)

`IDLE` → `DISCUSSING` (참여자 발화) → `SYNTHESIZING` (취합) → `VOTING` (찬반 + 조건부) → `AWAITING_USER` (사용자 결재 대기, autonomy=manual 일 때) → `APPLYING` (ExecutionService 적용) → `DONE`

분기:
- `FAILED` — 합의 실패 + maxRetries 초과
- `EXPIRED` — timeout (consensus_decision = 24 시간 default, R7 D4)
- `ABORTED` — 사용자 중단

#### 회의록 (R6 D3 + R10 Task 11 + R11 Task 8)

- 메타 헤더 — 참여자 / 주제 / SSM / 경과 / 투표
- LLM 요약 — `'summarize'` capability 가진 첫 provider fallback chain (R10 D7 / R11 Task 9 정식)
- 비용 audit — provider 응답 token usage 추출 → `llm_cost_audit_log` append-only (R11 Task 8)
- USD 추정 — 사용자 입력 단가 × 토큰 (R11 D5, default 0)

### 2.6 결재 인박스 (ApprovalInbox) — 5 kind discriminated union

| kind | 발사 시점 | 결정 | timeout |
|------|----------|------|---------|
| `cli_permission` | CLI 가 파일 쓰기/명령 실행 요청 | accept / reject (one-time/always) | 5 분 (R7 D2) |
| `mode_transition` | 사용자가 autonomy 모드 변경 | accept / reject / **conditional** (advisory comment) | 즉시 |
| `consensus_decision` | 회의가 합의 결과 도출 | accept / reject / conditional | 24 시간 (R7 D4) |
| `review_outcome` | 외부 리뷰어 결과 반영 (R8+) | accept / rework / fail | — |
| `failure_report` | CLI 실패 / 회로 차단 등 (R9+) | acknowledge | — |
| `circuit_breaker` | 4 tripwire 발동 (R9 D2 / R10 Task 9) | acknowledge | — |

**conditional decision 처리 (R11 Task 10 / D7):**
- `cli_permission` / `consensus_decision` — comment 가 다음 회의 system message 로 prepend
- `mode_transition` — `pendingAdvisory` slot (in-memory) 에 저장 → 다음 회의 시작 시 system message append + slot 비움 (이중 주입 방지 latch)

**ApprovalInbox 분할 layout (R11 Task 7):**
- 좌측 list-pane (filter: pending/approved/rejected/all)
- 우측 detail-pane (md+ 420px) — 5 카드:
  - DetailHeader (kind + 상태 + 시각)
  - ImpactedFiles (cli_permission 의 파일 변경 목록, dryRunPreview)
  - DiffPreview (변경 내용 미리보기)
  - ConsensusContext (회의 투표 history)
  - ActionBar (accept / reject / conditional)

### 2.7 자율 모드 + 큐

3 모드:

| 모드 | 동작 |
|------|------|
| `manual` | 모든 결정 사용자 승인 필수 |
| `auto_toggle` | autonomy=true 일 때 cli_permission/consensus_decision 자동 accept |
| `queue` | 백그라운드 큐 — QueuePanel 에서 사용자가 라인별 검토/실행 |

`AutonomyConfirmDialog` — `manual → auto_toggle` / `manual → queue` 두 전환만 확인 (R9 D4).

#### 4 tripwire Circuit Breaker (R9 + R10 Task 9 persist)

| tripwire | 의미 |
|----------|------|
| `files_per_turn` | 단일 턴 내 파일 수 (default 20+) |
| `cumulative_cli_ms` | CLI 누적 실행 시간 (default 30 분+) |
| `queue_streak` | queue 연속 실패 |
| `same_error` | 동일 에러 연속 |

발동 시 → autonomy → manual 자동 다운그레이드 + circuit_breaker approval row 생성 + force=true OS 알림. R10 마이그레이션 012 (`circuit_breaker_state`) 가 재시작 후에도 counter 보존 (D6).

### 2.8 알림 (Notification) — kind 별 prefs

| kind | OS toast | App badge | 음성 |
|------|---------|-----------|------|
| `approval_created` | ✅ | ✅ | 옵션 |
| `approval_decided` | ✅ | — | — |
| `meeting_completed` | ✅ | ✅ | 옵션 |
| `circuit_breaker_fired` | ✅ (force) | ✅ | 옵션 |
| (기타) | 사용자 prefs | 사용자 prefs | — |

부팅 시 seed (R9 D6 — INSERT OR IGNORE).
locale-aware (R11 D9) — `notification-labels.ts` dictionary, 한국어 default + 영어 번역, LanguageTab 변경 시 즉시 반영.

### 2.9 권한 시스템 — 3 mode × 3 CLI × 3 project kind

`PermissionFlagBuilder` (R10 Task 5):

| autonomyMode | 동작 |
|--------------|------|
| `auto` | CLI 에 모든 권한 위임 (`--dangerously-allow-all` 등) |
| `hybrid` | 매 turn 사용자 승인 게이트 (turn-level approval) |
| `approval` | strict path allowlist + 매 turn 승인 |

CLI 별 자동 승인 플래그 매트릭스는 `docs/specs/appendix/cli-matrix.md` (R1 land).

`external` project + `auto` 조합은 비활성 (path-guard 안전 보장 어려움, CA-3).

### 2.10 첫 부팅 Onboarding (R11 Task 6)

5-step wizard. first-boot 자동 진입 (`completed=false` 시 view='onboarding' 강제).

| Step | 화면 |
|------|------|
| 1 | Welcome (인사 + 앱 소개) |
| 2 | Staff (직원 선택) — provider:detect IPC 로 자동 + 사용자 추가 |
| 3 | RoleAssignment — provider 별 displayName / role / personality / expertise 한 줄 입력 |
| 4 | Permissions — 3 mode 라디오 (default `hybrid`) |
| 5 | FirstProject — kind 선택 (new/external/imported) + slug + 생성 |

완료 후 Dashboard 자동 진입 + reentry 안 함. Settings.AboutTab "온보딩 다시 시작" CTA 정식 wire (`completed=false` + `currentStep=1` reset).

### 2.11 Settings (10 탭, R10 Task 6)

horizontal Radix Tabs (R10 D3 — vertical sidebar nav 각하).

| 탭 | 용도 |
|----|------|
| Members | 직원 목록 + 편집 진입 |
| Notifications | kind 별 prefs (toast / badge / 음성) |
| AutonomyDefaults | 자율 모드 default + LLM 비용 가시화 (R11 Task 8) |
| ApiKeys | API provider 키 (safeStorage) |
| Theme | 6 테마 선택 |
| Language | ko / en + 3-step 동시 (`i18n.changeLanguage` + `config:update-settings` + `notification:set-locale`) |
| Path | ArenaRoot 경로 + project kind default |
| Cli | CLI 자동 승인 플래그 매트릭스 |
| Security | safeStorage 상태 + 권한 모드 |
| About | 버전 + 라이선스 + Onboarding CTA |

## 3. 6 테마 시스템

3 family × 2 mode = 6 테마:

| family | 특징 |
|--------|------|
| Warm | 둥근 박스 + 부드러운 색 + 여유 spacing |
| Tactical | 12 분절 게이지 + 코너 브라켓 + 시안 / 오렌지 액센트 |
| Retro | ASCII 프레임 + mono-prefix + 한국어 터미널 스타일 (`$ 채널`) |

themeKey 3-way DOM 분기 — 같은 컴포넌트에서 `themeKey === 'retro'` 등 if 분기 (R5 D3). 같은 색만 스왑하는 cosmetic theme 이 아니라 **형태 자체가 분기**.

R10 Task 7 형태 토큰 정식 wire — `panelClip` / `cardTitleStyle` / `miniBtnStyle` / `gaugeGlow` / `avatarShape` 5 종.

상세: [`docs/design/README.md`](디자인/README.md).

## 4. 자율 작업 큐 (QueuePanel)

사용자가 AI 에 작업을 라인 단위로 위임 — 백그라운드 실행.

- 라인 추가 (`use-queue.addLines`) — Optimistic UI (R10 D8)

- active spotlight — 현재 실행 중 라인 강조

- drag-and-drop 순서 변경 — HTML5 native + React state (R9 D3, `@dnd-kit/core` X)

## 5. 보안 + 데이터 보호

### 5.1 ArenaRoot 봉인 (path-guard)

모든 파일 쓰기 = `<ArenaRoot>/` 또는 `<ArenaRoot>/avatars/` 안. 외부 쓰기 절대 금지. external project 의 junction 너머는 realpath 비교 (CA-3 TOCTOU).

### 5.2 secrets 암호화

API 키 등 민감 데이터는 Electron `safeStorage.encryptString` (macOS Keychain / Windows DPAPI / Linux libsecret 자동 위임). Renderer 에 평문 노출 금지.

### 5.3 셸 명령 안전

`execFile` + `shell: false` + 구조화 `CommandRequest`. 셸 문자열 실행 절대 금지.

### 5.4 ExecutionService 단일 진입점

모든 파일 변경 = `dryRun → 사용자 승인 → atomic apply → rollback` 순서. 감사 로그 (`AuditEntry`) 영구 기록.
