# WSL CLI Detection & Execution Design

## Problem

Electron 앱이 Windows에서 실행되지만, Claude Code 등 CLI 도구가 WSL 내부에만 설치되어 있을 수 있다. 현재 `where` 명령으로만 감지하므로 WSL CLI는 발견되지 않는다.

## Approach: `wsl.exe` Wrapping

기존 `resolveWindowsCommand` 패턴(`.cmd` → `cmd.exe /C`, `.ps1` → `pwsh.exe`)과 동일하게, WSL CLI는 `wsl.exe -d <distro> -- <command> ...args`로 래핑한다.

stdin/stdout 파이프는 `wsl.exe` 래핑 시에도 그대로 작동하므로 스트리밍 코드 변경 없음.

## Changes

### 1. Type Extensions

**`DetectedCli`** (`src/shared/ipc-types.ts`):
- `wslDistro?: string` — `undefined` = native, `string` = WSL distro name

**`CliProviderConfig`** (`src/shared/provider-types.ts`):
- `wslDistro?: string` — DB 저장/복원 시 어떤 distro에서 실행할지 기억

**`SettingsConfig`** (`src/shared/config-types.ts`):
- `preferredWslDistro?: string` — 사용자 지정 distro (없으면 기본 distro 사용)

### 2. Detection (`cli-detect-handler.ts`)

1. 기존 `where` 감지 (Windows native) 시도
2. Native로 못 찾은 CLI에 대해:
   - `wsl.exe -l -q`로 사용 가능한 distro 목록 조회
   - `preferredWslDistro` 설정 있으면 해당 distro만, 없으면 기본 distro
   - `wsl.exe -d <distro> -- which <command>` 실행
   - 찾으면 `wslDistro` 포함한 `DetectedCli` 반환

### 3. Execution (`cli-process.ts`)

`resolveWindowsCommand` 확장:
- `wslDistro` 파라미터 추가
- `wslDistro`가 있으면 다른 Windows 해석 전에 `wsl.exe -d <distro> --` 래핑

### 4. Factory (`factory.ts`)

`getRuntimeCliConfig`에서 `wslDistro`를 runtime config에 전달.

### 5. Unchanged

- `CliProvider`, `CliOutputParser`, `CliProcessManager` 스트리밍 로직 — 변경 없음
- `CliSessionState` — 변경 없음
- 모든 CLI config (claude/gemini/codex) — 변경 없음
