# 회의 시스템 새로 만들기 — 1차 작업 쉬운 설명

> 본 문서는 코드 모르는 사용자를 위한 사무실 메타포 설명입니다.
> 코드명 = `R12-C2 T10a` (`P2-4a` 단계 첫 sub-task)
> 위치 = worktree `feat/r12-c2-meeting-redesign`

---

## 1. 지금 뭐 하려는지

**한 줄로**: 직원들이 모여서 회의하는 *진행 엔진* 을 통째 새로 만든다.

비유하자면 — 사무실에 회의실이 있는데, 옛날에는 회의 진행자가 정해진 12 단계 절차표 (`OPINION_GATHERING` → `OPINION_TALLY` → `AGREEMENT_VOTE` → ...) 를 들고 다니면서 "지금 N단계니까 다음은 N+1 단계" 식으로 진행했음. 너무 복잡해서 폐기.

새 진행자는 훨씬 단순한 6 단계만 본다.

---

## 2. 왜 새로 만드는지

옛 12 단계 절차표는:
- 한 회의 안에서도 분기가 너무 많아 (작업 모드 / 대화 모드 / 의견 분기 / 투표 분기 / 실행 분기 ...) 진행자 코드가 904 + 815 + 417 = 2,136 줄로 비대화됨.
- 부서별로 흐름이 달라야 하는데 한 절차표에 다 욱여넣어서 "기획 부서 회의" 와 "리뷰 부서 회의" 가 같은 절차를 따라가는 부자연스러움.
- 의견이 무한 누적되는 "끝말잇기" 자연 무한 주제에 약함.

새 6 단계는:
- 단순하다 (의견 → 취합 → 일괄 투표 → 자유 토론 → 회의록 → 인계).
- 부서 5종 (기획 / 리뷰 / 검토 / 디자인 / 구현 / 일반) 이 같은 진행 엔진 위에 *얹기만* 하면 됨.
- 의견 트리 깊이 3 + 라운드 cap 두 가지 안전장치로 무한 회의 차단.

## 3. 새 회의 흐름 — 6 단계

```
   ┌──────────────────────────────────────────────────────────────┐
   │  회의 시작                                                    │
   └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  1. 의견 제시 (gather)                                        │
   │  직원 모두에게 "의견 적어 와" 1 턴씩 돌림.                    │
   │  각자 발화 ID 받음 (codex_1, claude_1, gemini_1).             │
   │  같은 직원이 의견 여러 개 적어도 됨.                          │
   └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  2. 시스템 취합 (tally)                                       │
   │  진행자가 모은 의견에 카드 번호 매김.                         │
   │    ITEM_001 / ITEM_002 / ITEM_003                             │
   │  (네트워크 호출 X, 즉시 끝남.)                                │
   └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  ★ 2.5. 일괄 동의 투표 (quick_vote)                           │
   │  의견 list 한꺼번에 직원들에게 보내고 동의/반대 투표 받음.    │
   │                                                              │
   │  → 만장일치 의견 = 즉시 합의. (자유 토론 skip)                │
   │  → 만장일치 못 받은 의견 = 다음 단계로.                       │
   └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  3. 자유 토론 (free_discussion)                               │
   │  진행자가 의견 1 개씩 꺼내서 "이 의견 어떻게 생각해" 물음.    │
   │  직원들이:                                                    │
   │    - 동의 / 반대 / 보류                                       │
   │    - 또는 수정안 / 반대안 / 보강안 (자식 카드 등록)           │
   │      예: ITEM_002 의 수정안 = ITEM_002_01                     │
   │      그 수정안의 또 수정안 = ITEM_002_01_01 (깊이 3 한계)     │
   │  합의 도달 (모두 agree) → 다음 의견                           │
   └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  5. 회의록 작성 (compose_minutes)                             │
   │  모더레이터 직원이 회의 통째 받아 회의록 markdown 작성.       │
   │    [합의 항목] — 합의된 의견 통째 (truncate 금지)             │
   │    [제외 항목] — 제외된 의견 통째 + 제외 사유                 │
   │  파일 저장: <ArenaRoot>/<projectId>/consensus/                │
   │           <meetingId>/minutes.md                              │
   └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌──────────────────────────────────────────────────────────────┐
   │  6. 인계 (handoff)                                            │
   │  채널 설정 보고 분기:                                         │
   │    - 자동 인계 (auto)  = 다음 부서로 즉시 보냄                │
   │    - 사용자 승인 (check) = Notification + 일시정지            │
   │                            "회의 끝남, 다음 부서 보낼까요?"   │
   └──────────────────────────────────────────────────────────────┘
                          │
                          ▼
                       회의 종료
```

> **단계 4 가 빠진 이유**: spec §5 가 "step 5" 까지 6 단계로 매겨져 있어서 그대로 따름. 단계 4 = "다음 의견으로 넘어가서 step 3 반복" — 별도 phase 가 아니라 step 3 의 loop 내부.

## 3-1. 부서별 흐름 차이 — 위 6 단계는 *풀세트 부서* 만

위 6 단계 흐름은 **풀세트 부서 (기획 / 리뷰 / 검토)** 가 그대로 따른다. 부서마다 회의 흐름이 달라서 일부는 단계가 빠지거나 아예 회의를 안 한다.

### 아이디어 부서 (idea) — D-B-Light, 자유토론 X

```
   ┌─────────────────────────────────────────────────────────┐
   │  1. 의견 제시 (gather)                                   │
   │  직원들이 아이디어 카드 적어 옴.                         │
   │  같은 직원이 여러 개 적어도 됨.                          │
   └─────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌─────────────────────────────────────────────────────────┐
   │  2. 시스템 취합 (tally)                                  │
   │  카드에 ITEM_001 / ITEM_002 ... 번호 붙임.              │
   │  (트리 X — root 의견만, 단순 list)                      │
   └─────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌─────────────────────────────────────────────────────────┐
   │  ★ USER_PICK (사용자 단계, 직원 X)                       │
   │  사용자가 카드 list 보고 마음에 드는 것 *고름* +         │
   │  자유 코멘트 작성.                                       │
   │  (직원들 일괄 투표 X / 자유 토론 X / 합의 X)            │
   └─────────────────────────────────────────────────────────┘
                          │
                          ▼
   ┌─────────────────────────────────────────────────────────┐
   │  6. 인계 (handoff)                                       │
   │  사용자가 고른 카드 + 코멘트 → 기획 부서로 자동 인계.   │
   └─────────────────────────────────────────────────────────┘
```

**핵심**:
- 직원들이 *자유 토론 안 함*. 한 번 의견 적어 오고 끝.
- 일괄 투표도 안 함 — 직원끼리 합의 X.
- *사용자가 진행자 역할* — 카드 보고 골라서 다음 부서로.
- 회의록 작성 X. 사용자 선택 결과 + 코멘트가 곧 인계 payload.

> 비유: 브레인스토밍 회의실에서 직원들이 포스트잇 잔뜩 붙이고 나간 다음, 사장이 마음에 드는 것 골라서 기획팀에 넘기는 모델.

### 기획 / 리뷰 / 검토 부서 — 풀세트 (위 6 단계 전부)

위 §3 의 6 단계 흐름 그대로. 자유 토론에서 직원들이 의견 트리 (수정안 / 반대안 / 보강안) 만들면서 합의 도달까지 라운드 누적.

### 디자인 부서 — 7 단계 (풀세트 회의 2 회)

기획 회의처럼 풀세트지만 *두 번* 한다.
1. 와이어프레임 회의 (구조도) — 풀세트 6 단계
2. 디자인 회의 (HTML/CSS) — 풀세트 6 단계

각 회의 끝에 Playwright 가 desktop / mobile 화면 PNG 자동 생성.

### 구현 부서 (implement) — 회의 X

회의 자체를 안 한다. designated 직원 1 명이 spec 받아 코드 작성. 1 차 backend 시점에는 simple 1 명 모델, 본격 분담 + worktree 분할은 R12-W (다른 phase).

### 일반 부서 (general) — 잡담 채널, 회의 X

회의 X. `[##본문]` 으로 감싼 메시지만 의견 카드 등록 + 가벼운 동의/반대 카운터. 합의 X / 회의록 X / 인계 X.

---

### 1 차 (T10a) 가 영향 주는 부서

| 부서 | 1 차 backend 사용 단계 |
|------|---------------------|
| **아이디어** | `gather` → `tally` → `handoff` (3 phase 만 사용. 자유 토론 / 회의록 phase 호출 X) |
| **기획 / 리뷰 / 검토** | 6 phase 전부 사용 (`gather` → `tally` → `quick_vote` → `free_discussion` → `compose_minutes` → `handoff`) |
| **디자인** | 풀세트 회의 2 회 — 6 phase 두 번 돌림 |
| **구현** | 본 backend 호출 X (회의 X) |
| **일반** | 본 backend 호출 X (회의 X, opinion 테이블만 [##] 카드 등록 용도로 공유) |

→ 1 차 backend 가 *모든 풀세트 + Light 부서가 공유하는 토대*. 부서별 차이는 어느 phase 까지 진행할지 결정 (USER_PICK 진입 자체는 P3 책임 — 1 차에는 phase enum + handoff trigger 까지만).

## 4. 본 1차 작업이 어디까지

이번 1차 (T10a) 는 **backend 만**. 화면 박스 (회의 진행 보여주는 UI) 는 다음 단계 (P3) 책임.

| 영역 | 1차 작업 | 다음 단계 |
|------|---------|----------|
| 진행 엔진 (orchestrator + turn-executor + session) 새 6 단계 | ✅ 본 1차 | — |
| 직원 응답 JSON 양식 4 종 (zod 스키마) | ✅ 본 1차 | — |
| 회의록 파일 저장 (T9 에서 완성한 서비스 호출) | ✅ 본 1차 | — |
| 외부 명령 통로 (회의 시작 / 회의 중단) | ✅ 본 1차 | — |
| 화면 박스 layout (SsmBox — 부서별 다른 모양) | ❌ | P3 |
| 채팅창 카드 (회의 중 의견 발화 표시) | ❌ | P3 |
| 채널 설정 모달 (인계 방식 / 라운드 cap 토글) | ❌ | P3 또는 P6 |

→ 1차 끝 시점 = 옛 진행 엔진 파일은 살아있지만 *아무도 호출 안 함* (dead code). T10b 가 통째 삭제.

## 5. 사용자가 결정해줘야 하는 4 항목

설계 게이트 [1] 에서 사용자 OK 받기로 한 항목들. 어려운 부분만 쉽게 풀어 씀.

### ① 화면 신호 이름 — "회의가 다음 단계로 넘어갔어요"

**상황**: 화면 박스 (회의 진행 보여주는 박스) 가 현재 회의가 어느 단계인지 알아야 그림을 그림. 진행 엔진이 "지금 단계 바뀜" 신호를 화면으로 쏨.

**옵션**:
- (가) 새 신호 1 종 추가 — `회의-단계-바뀜` (새 6 단계 이름이 들어감)
- (나) 옛 신호 (`회의-상태-바뀜`) 그대로 쓰는데 안에 들어가는 값만 새 6 단계 이름으로 갱신

**추천 (제안)**: (가) + (나) 둘 다. 새 신호 추가 + 옛 신호도 새 6 단계 이름 dispatch (P3 화면 박스 마이그레이션 동안 둘 다 받을 수 있게).

### ② 직원 발화 번호 (codex_1, claude_2) 어디서 셀까

**상황**: 한 회의 안에서 직원별로 1, 2, 3 번 매기고 회의 끝나면 리셋. 다음 회의에서 다시 1 부터.

**옵션**:
- (가) 회의 진행 엔진 내부 카운터 메모리에 들고 있기
- (나) 회의록 DB (의견 row 의 `author_label`) 가 진실 source — 매번 거기서 다음 번호 물어보기

**추천 (제안)**: (나). DB 가 진실. 매번 `OpinionService.nextLabelHint(meetingId, providerId)` 호출 (T8 에서 이미 만들어 둔 method). 앱 재시작해도 번호 안 꼬임.

### ③ 회의 끝나면 다음 부서로 어떻게 인계?

**상황**: 회의 끝났을 때 자동 인계할지 (auto), 사용자 승인 받을지 (check) 결정.

**옵션**: spec §11.14 의 채널 설정 두 토글 중 하나:
- 자동 인계 (auto)
- 사용자 승인 후 인계 (check)  ◯ ← spec 표기 디폴트

**추천 (제안)**: 디폴트 = `check` (사용자 승인). 채널마다 사용자가 모달에서 바꿀 수 있음 — 그건 P6 작업. 본 1차 = 컬럼 *읽기* 만, 모달 자체는 X.

### ④ 회의가 끝없이 돌아가지 않게 막는 안전장치

**여기서 "자유 토론" = 풀세트 부서 (기획 / 리뷰 / 검토 / 디자인) 의 step 3 자유 토론만**.

| 부서 | 자유 토론 단계 | ④ 안전장치 적용 |
|------|---------------|----------------|
| 아이디어 (idea) | 없음 (D-B-Light, 사용자가 카드 골라서 끝) | 무관 |
| 기획 / 리뷰 / 검토 | 있음 (step 3) | **적용** |
| 디자인 (design) | 있음 (와이어프레임 회의 + 디자인 회의 각각 step 3) | **적용** |
| 구현 (implement) | 회의 X | 무관 |
| 일반 (general) | 회의 X (잡담) | 무관 |

**상황**: 풀세트 부서의 자유 토론에서 직원들이 끝없이 수정안 / 반대안 내면 회의가 무한히 돌아감. 두 가지 안전장치로 막음.

**두 안전장치**:
- (a) **채널별 라운드 cap** (`channels.max_rounds`) — 사용자가 채널 설정에서 정함. 예: "5 라운드 도달 시 사용자 호출". NULL 이면 무제한.
- (b) **하드 cap = 5** (`maxConversationRounds`) — D-A 라운드 5 에서 이미 land 한 자연 무한 주제 차단 (끝말잇기 같은 거). 사용자 설정 X, 시스템이 강제.

**라운드 1 회 단위** = step 3 안에서 진행 의견 1 개에 대해 직원 N 명이 한 바퀴 돌며 동의/반대/수정 응답한 묶음. 의견 1 개 합의되면 다음 의견으로 넘어가고 라운드 카운터 리셋.

**추천 (제안)**: 둘 다 존중. (a) 가 NULL 이어도 (b) 가 5 라운드에서 자동 멈춤. 사용자가 (a) 를 3 으로 설정하면 3 라운드에서 멈춤 (둘 중 빠른 쪽).

> **별개 안전장치 (참고)**: 의견 트리 깊이 cap = 3 (`ITEM_001` → `ITEM_001_01` → `ITEM_001_01_01` 까지). 이건 라운드 cap 과 다른 차원 — 라운드 cap = 시간 축 (몇 번 돌렸나), 깊이 cap = 공간 축 (수정안의 수정안의 수정안). 풀세트 부서 자유 토론 step 3 에서 둘 다 동시 작동.

## 6. 1차 작업 끝 검증

코드가 제대로 됐는지 확인할 4 가지:

1. **새 진행 엔진이 옛 절차표 호출 안 함** — `grep` 으로 확인. 옛 클래스 (`ConsensusStateMachine`, `sessionMachine`, `SessionState`) 이름이 새 진행 엔진 파일 안에 0 회 등장.
2. **타입 에러 0** — `npm run typecheck` 통과.
