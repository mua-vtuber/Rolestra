# AI Chat Arena v2 — Coding Standards

작성일: 2026-02-13

## 1. 기본 원칙

- 기능보다 규칙 우선: 규칙 위반 구현은 병합하지 않는다.
- 작은 변경 단위: 한 PR은 한 주제만 다룬다.
- 추상화는 필요 최소만: 과한 일반화 금지.

## 2. TypeScript/구조 규칙

- `strict` 모드 유지.
- `any` 사용 금지(불가피할 경우 주석으로 이유 명시).
- 공용 타입은 `src/shared`에 둔다.
- provider/file/db/ipc 계층 경계를 넘는 직접 호출 금지.

### Import 규칙
- `renderer` → `main` 직접 import 금지 (반드시 IPC 경유).
- `main` → `renderer` 직접 import 금지.
- `main`/`renderer` 모두 → `shared`는 허용.
- `preload` → `shared`만 허용 (main/renderer 직접 참조 금지).
- eslint `no-restricted-imports`로 위반을 자동 감지한다.

### 네이밍 컨벤션
- **파일명**: kebab-case (`consensus-machine.ts`, `turn-manager.ts`)
- **클래스/인터페이스/타입**: PascalCase (`ConsensusStateMachine`, `ProviderConfig`)
- **함수/변수**: camelCase (`getNextSpeaker`, `currentRound`)
- **상수/enum 값**: UPPER_SNAKE_CASE (`DISCUSSING`, `MAX_RETRIES`)
- **React 컴포넌트 파일**: PascalCase (`ChatView.tsx`, `MessageBubble.tsx`)
- **번역 키**: dot-separated lowercase (`chat.send`, `provider.status.ready`)

## 3. IPC 규칙

- 문자열 채널 직접 사용 금지, typed wrapper만 사용.
- IPC 페이로드는 타입으로 정의하고 변경 시 버전(`schemaVersion`) 고려.
- 개발 모드 런타임 검증(zod) 실패는 즉시 에러로 처리.

## 4. Provider 규칙

- Provider 분기는 capability 기반으로 처리.
- `ProviderConfig`는 discriminated union 유지.
- provider별 특화는 adapter 내부로 숨기고 외부 인터페이스는 공통 유지.

## 5. 실행/권한 규칙

- 파일 반영은 `ExecutionService`를 반드시 경유.
- `dry-run -> 승인 -> atomic apply -> rollback` 순서 고정.
- `runCommand`는 구조화 요청(`command`, `args`, `cwd`)만 허용.
- `exec`, `bash -c`, 임의 셸 문자열 실행 금지.

## 6. DB/마이그레이션 규칙

- forward-only, idempotent, startup blocking on failure.
- 스키마 변경 시 migration + repository + 타입 + 테스트 동시 수정.
- 분기/메모리 운영 컬럼은 제거 금지(하위호환 깨짐).

## 7. i18n/UI 규칙

- 사용자 노출 문자열 하드코딩 금지 (`t()` 사용 필수).
- 번역 키는 네임스페이스 구조 유지 (`chat.send`, `provider.status.ready` 등).
- 개발자 로그/내부 오류는 영어 고정 허용.

## 8. 에러/로그 규칙

- 사용자 메시지는 안전하고 간결하게, 내부 원인은 구조화 로그로 남긴다.
- 로그 필수 필드: `requestId`, `conversationId`, `component`, `action`, `result`.
- 민감정보(API 키, 토큰, 절대경로 일부)는 로그 마스킹.

## 9. 테스트 규칙

- 신규 로직은 최소 1개 단위 테스트 추가.
- 상태머신/권한/IPC는 회귀 테스트 필수.
- 버그 수정은 재현 테스트를 먼저 추가.

## 10. 금지 사항

- Renderer에서 Node API 직접 접근
- typed IPC 우회
- 승인 없는 실제 파일 반영
- 마이그레이션 파일 임의 수정(적용 이력 있는 파일)
- 하드코딩 UI 문자열 추가
