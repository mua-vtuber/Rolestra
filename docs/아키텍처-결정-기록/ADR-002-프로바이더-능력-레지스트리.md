# ADR-002: Provider는 Capability + Registry 기반으로 설계한다

- 상태: 승인
- 날짜: 2026-02-13
- 결정자: 프로젝트 팀
- 관련 문서: `docs/설계 보완안 (취합).md`, `docs/설계-문서.md`

## 배경

provider/모델을 코드에 하드코딩하면 신규 provider 추가, 모델 변경 시 재배포가 필요해진다.

## 결정

- 컴파일 타임: `ProviderConfig`를 discriminated union으로 정의
- 런타임: capability(`streaming`, `resume`, `tools`, `web-search` 등) 기반 분기
- provider/model 카탈로그는 registry 데이터로 관리

## 근거

- 타입 안전성과 런타임 확장성 동시 확보
- provider별 특수 처리를 adapter 내부로 캡슐화 가능

## 영향

- 긍정: 하드코딩 감소, 확장성 증가
- 부정: 초기 구조 설계 비용 증가
- 마이그레이션 영향: provider 설정 스키마 버전 관리 필요

## 대안

- enum + switch 기반 고정 분기: 단순하지만 장기 유지보수 비용이 큼.
