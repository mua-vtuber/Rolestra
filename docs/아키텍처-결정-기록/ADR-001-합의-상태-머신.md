# ADR-001: 합의 엔진은 상태머신으로 구현한다

- 상태: 승인
- 날짜: 2026-02-13
- 결정자: 프로젝트 팀
- 관련 문서: `docs/설계 보완안 (취합).md`, `docs/설계-문서.md`

## 배경

작업 모드의 합의 흐름(토론/취합/동의/승인/실행)은 상태 전이가 많고 예외 케이스가 많다.
암묵적 분기 로직으로 구현하면 재개/복구/디버깅이 어렵다.

## 결정

합의 엔진은 `ConsensusStateMachine`으로 구현한다.

- 상태: `DISCUSSING`, `SYNTHESIZING`, `VOTING`, `AWAITING_USER`, `APPLYING`, `DONE`, `FAILED`
- 이벤트/가드/타임아웃/재시도 상한(`maxRetries`) 명시
- 취합자 선택 전략(`aggregatorStrategy`): `strongest`(기본), `last-speaker`, `designated`, `round-robin`
- 상태 스냅샷 저장(전이 시점 + 주기 체크포인트 + 종료 시 flush)

## 근거

- 복잡한 흐름을 명시적 전이로 관리 가능
- 예외/복구 경로가 코드로 드러남
- 테스트 가능성 증가

## 영향

- 긍정: 안정성/가시성/회귀 방지 향상
- 부정: 초기 구현 복잡도 증가
- 마이그레이션 영향: 상태 스냅샷 저장 스키마 필요

## 대안

- 일반 서비스 로직 내 if/else 전개: 초기 구현은 빠르지만 유지보수 리스크가 큼.
