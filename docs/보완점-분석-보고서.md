## 2. 버그 및 로직 오류 — **[전체 해결됨]**

### 2.1 `handleToggleSearch`가 항상 닫기만 수행 — **[해결됨]**

- **파일**: `src/renderer/hooks/useChat.ts:508-511`
- **내용**:
  ```typescript
  const handleToggleSearch = useCallback((): void => {
    setSearchOpen(false);   // 항상 false — 토글이 아님!
    setSearchQuery('');
  }, []);
  ```
- **영향**: 함수명과 동작이 불일치. 열기 기능이 `Ctrl+F` 키보드에만 의존
- **권장**: `setSearchOpen(prev => !prev)` 로 수정

### 2.2 `requestedRoundSetting`이 새 대화 시 초기화되지 않음 — **[해결됨]**

- **파일**: `src/main/ipc/handlers/chat-handler.ts:90`
- **내용**: `setActiveSession(null)` 호출 시 `requestedRoundSetting`은 리셋되지 않음. 이전 대화의 라운드 설정이 새 대화에 무단 상속됨
- **권장**: `setActiveSession(null)` 시 `requestedRoundSetting = 'unlimited'` 초기화 추가

### 2.3 `useEffect` 의존성 배열 위반 (stale closure) — **[해결됨]**

- **파일**: `src/renderer/hooks/useChat.ts:170-173`
- **내용**:
  ```typescript
  useEffect(() => {
    void window.arena.invoke('chat:set-rounds', { rounds });
    // eslint-disable-next-line react-hooks/exhaustive-deps
  }, []);
  ```
  `rounds` 값이 클로저에 캡처되지만 의존성 배열에서 제외 → 항상 초기값 `1`만 전송
- **권장**: `[rounds]`로 의존성 수정 또는 의도된 동작이면 주석으로 설명

### 2.4 `totalTokens` 변수 선언 후 미사용 — **[해결됨]**

- **파일**: `src/main/engine/orchestrator.ts:650-653`
- **내용**: `const totalTokens = providerUsage?.totalTokens ?? null;` — 선언만 하고 사용 안 함
- **권장**: 제거하거나 token 사용량 추적에 활용

### 2.5 `interruptWithUserMessage()` 메서드 빈 구현 — **[해결됨]**

- **파일**: `src/main/engine/turn-manager.ts:147-156`
- **내용**: 메서드 본문이 주석뿐이고 실제 로직 없음. `Orchestrator.handleUserInterjection()`이 호출하지만 아무 동작 안 함
- **권장**: 실제 인터럽트 로직 구현 또는 불필요하면 인터페이스에서 제거

## 3. 동시성 / 경쟁 조건 — **[전체 해결됨]**

### 3.1 [High] `reenterLoop()`이 동시에 실행될 수 있음 — **[해결됨]**

- **파일**: `src/main/engine/orchestrator.ts:197-205`
- **내용**: `onPhaseChange` 리스너가 `DISAGREE`와 `USER_REVISE` 이벤트로 빠르게 연속 호출되면, `reenterLoop()` → `this.loop()`이 두 번 동시 실행됨. `TurnManager`의 `speakerIndex`, `turnsInCurrentRound`가 동시 변이
- **권장**: `this.running` 플래그 외에 mutex/세마포어 패턴 적용. 또는 `reenterLoop` 호출을 큐잉

### 3.2 [High] `chat-handler.ts` 모듈 레벨 싱글톤 경쟁 — **[해결됨]**

- **파일**: `src/main/ipc/handlers/chat-handler.ts:40-49`
- **내용**:
  ```typescript
  let activeSession: ConversationSession | null = null;
  let activeOrchestrator: ConversationOrchestrator | null = null;
  ```
  IPC 핸들러가 빠르게 연속 호출되면 (예: `chat:send` 더블 클릭) `activeOrchestrator` 체크와 할당 사이에 레이스 컨디션 발생 → 두 개의 Orchestrator가 동일 세션에서 실행 가능
- **권장**: 원자적 세션 전환 로직 구현 (예: `isBusy` 플래그 + early return)

### 3.3 [Medium] `pendingPatches`가 대화 전환 시 비워지지 않음 — **[해결됨]**

- **파일**: `src/main/ipc/handlers/execution-handler.ts:47`
- **내용**: `setActiveSession(null)` 호출 시 `pendingPatches` Map이 클리어되지 않음. 이전 대화의 패치를 새 대화에서 승인할 수 있음
- **권장**: 세션 전환 시 `pendingPatches.clear()` 호출

### 3.4 [Medium] `killProcess()` 강제 종료 타이머가 새 프로세스를 타겟할 수 있음 — **[해결됨]**

- **파일**: `src/main/providers/cli/cli-provider.ts:486-497`
- **내용**: `this.process = null`이 즉시 실행되지만, 3초 후 `forceKillTimer` 콜백에서 `this.process`를 다시 참조 → 3초 내에 새 프로세스가 생성되면 **잘못된 프로세스에 SIGKILL** 전송
- **권장**: 로컬 변수에 프로세스 참조를 캡처: `const proc = this.process;`

## 4. 메모리 누수 / 리소스 정리 — **[전체 해결됨]**

### 4.1 [High] `_executionApprovalResolve`가 `stop()` 시 해제되지 않음 — **[해결됨]**

- **파일**: `src/main/engine/orchestrator.ts:128-136`
- **내용**: `stop()` 메서드에서 `_consensusResolve`는 정리하지만 `_executionApprovalResolve`는 미정리 → 좀비 프로미스 누수
- **권장**: `stop()`에 `this._executionApprovalResolve?.(); this._executionApprovalResolve = null;` 추가

### 4.2 [Medium] `ConsensusStateMachine._snapshots` 무한 증가 — **[해결됨]**

- **파일**: `src/main/engine/consensus-machine.ts:568`
- **내용**: `this._snapshots.push(snapshot)` — 프루닝 로직 없음. 긴 세션에서 모든 전이 기록이 메모리에 누적
- **권장**: 최대 크기 제한 (예: 최근 100개 유지) 또는 오래된 스냅샷 DB 이관

### 4.3 [Medium] CSM 타이머가 폐기된 세션에서 발화 — **[해결됨]**

- **파일**: `src/main/engine/consensus-machine.ts:429-438`
- **내용**: `dispose()`가 Orchestrator/ConversationSession에서 호출되지 않음. 세션 전환 시 이전 CSM의 `phaseTimeout`이 여전히 동작 → 파괴된 `webContents`에 이벤트 발행 시도
- **권장**: 세션 정리 시 `csm.dispose()` 명시적 호출

### 4.4 [Low] `delay()` 타이머가 취소 불가능 — **[해결됨]**

- **파일**: `src/main/engine/orchestrator.ts:878-880`
- **내용**: `stop()` 호출 시 진행 중인 `setTimeout`이 취소되지 않음 (최대 2초 잔존)
- **권장**: `AbortSignal` 연동 `delay()` 구현

## 5. 에러 처리 미흡 — **[대부분 해결됨]**

> SSE 파서에 Zod 스키마 검증 적용 (`sse-schemas.ts`), 빈 catch 블록 0건. 에러 타입 분류 후 AbortError/API-level 에러 전파.

### 5.1 DB 쓰기 실패 무시 — **[해결됨]**

- **파일**: `src/main/engine/orchestrator.ts:337-349`
- **내용**: `generateConsensusDocument()`에서 DB 쓰기 실패가 빈 `catch {}` 블록으로 무시됨
- **권장**: 최소 `console.warn` + `StructuredLogger` 기록

### 5.2 SSE 파싱 실패 무시 — **[해결됨]** Zod 스키마 적용, 에러 타입 분류

- **파일**: `src/main/providers/api/api-provider.ts:315, 383, 439`
- **내용**: `parseSSE`, `parseAnthropicSSE`, `parseGoogleSSE`에서 `catch { /* skip */ }` — rate limit(429) 등 중요한 에러도 무시됨
- **권장**: 에러 타입 분류 후 `AbortError`, API-level 에러는 전파

### 5.3 `pause()` 실패 무시 — **[해결됨]**

- **파일**: `src/renderer/hooks/useChat.ts:238, 245`
- **내용**: `void pause().catch(() => undefined)` — 일시정지 실패 시 UI는 "diff 대기 중" 표시하지만 실제로 일시정지되지 않음
- **권장**: 실패 시 사용자에게 에러 표시 (Toast)

### 5.4 `reenterLoop()` 실패로 CSM 교착 — **[해결됨]**

- **파일**: `src/main/engine/orchestrator.ts:199-205`
- **내용**: `reenterLoop()` 실패가 `console.error`만 하고 CSM을 `FAILED`로 전이하지 않음 → CSM이 `DISCUSSING` 상태에서 영구 고착
- **권장**: catch 블록에서 `csm.transition('ERROR')` 호출

### 5.5 CLI Provider `stdin.write` 실패 무시 — **[해결됨]**

- **파일**: `src/main/providers/cli/cli-provider.ts:681`
- **내용**: persistent 모드에서 stdin 쓰기 실패 시 에러 정보 없이 프로세스 재시작
- **권장**: 에러 로깅 추가
