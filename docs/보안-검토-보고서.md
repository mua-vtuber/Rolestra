## CRITICAL (즉시 수정 필요)

### C1. IPC 입력 검증이 프로덕션에서 비활성화됨 — **[해결됨]** 크리티컬 채널에 zod 검증 추가, SSE 스키마 도입

**파일:** `src/main/ipc/router.ts:134-142`

```typescript
const isDev = process.env.NODE_ENV === 'development' ||
  !!process.env.ELECTRON_RENDERER_URL;
if (isDev) {
  await validateMetaDev(envelope.meta);
}
```

Zod 런타임 검증이 개발 모드에서만 수행됩니다. 프로덕션에서는 모든 IPC 페이로드(`envelope.data`)가 **무검증** 상태로 핸들러에 전달됩니다. `chat:send`, `config:set-secret`, `workspace:init`, `execution:approve` 등 민감한 채널 포함.

**영향:** 손상된 렌더러 프로세스가 임의의 페이로드를 모든 핸들러에 전달 가능.

**권장 수정:** 모든 IPC 채널에 Zod 스키마 검증을 프로덕션에서도 적용. 최소한 `config:set-secret`, `workspace:init`, `execution:approve/reject` 등 크리티컬 채널은 필수.

### C2. PatchApplier — 경로 경계 검증 없음 (Path Traversal) — **[해결됨]** PatchApplier에서 PermissionService 경로 검증 호출 추가

**파일:** `src/main/execution/patch-applier.ts:118-131`

```typescript
private applyCreate(entry: PatchEntry): void {
  if (fs.existsSync(entry.targetPath)) { ... }
  const dir = path.dirname(entry.targetPath);
  fs.mkdirSync(dir, { recursive: true });
  fs.writeFileSync(entry.targetPath, entry.newContent, 'utf-8');
}
```

`entry.targetPath`에 대한 **경로 검증이 전혀 없습니다.** `../../etc/cron.d/malicious`나 절대경로 `/etc/passwd`도 그대로 수행됩니다. `PermissionService`가 존재하지만 `PatchApplier`/`ExecutionService`에서 호출되지 않습니다.

**영향:** AI가 제안한 PatchSet의 `targetPath`가 프로젝트 폴더 외부 파일을 생성/수정/삭제할 수 있음.

**권장 수정:** `PatchApplier.applyEntry()`에서 반드시 `PermissionService.validateAccess()` 또는 동등한 워크스페이스 경계 검사를 수행.

### C4. provider-handler에서 No-op 암호화 어댑터 사용 — **[해결됨]** 독립 인스턴스 제거, 공유 `getConfigService()` 싱글턴 사용

### H3. `blockedPatterns`가 불완전하고 ReDoS 위험 존재 — **[해결됨]** RegExp 사전 컴파일, 패턴 보강, 사용자 입력 패턴 금지

**파일:** `src/main/execution/command-runner.ts:45-48`
**파일:** `src/shared/execution-types.ts:114-121`

```typescript
for (const pattern of this.policy.blockedPatterns) {
  if (new RegExp(pattern).test(fullCommand)) { ... }
}
```

- `rm\\s+-rf\\s+/`는 `rm -rf /`만 차단, `rm -rf /home`이나 `rm -rf ~`는 통과
- `new RegExp(pattern)`이 매 실행마다 컴파일되어 ReDoS 위험
- 사용자 정의 정책이 유입될 경우 `(a+)+$` 같은 패턴으로 메인 프로세스 행 가능

**권장 수정:** (a) 패턴을 RegExp 리터럴로 사전 컴파일 (b) 명령어별 인자 수준 허용 정책으로 전환 (c) 사용자 입력 패턴 금지.

### H4. Windows에서 `.cmd/.bat/.ps1` 실행 시 `shell: true` — **[해결됨]** `.cmd`/`.bat` 경로 화이트리스트 제한, 인자 이스케이프 추가

**파일:** `src/main/providers/cli/cli-provider.ts:49-57`

```typescript
function needsShell(command: string): boolean {
  if (process.platform !== 'win32') return false;
  const lower = command.toLowerCase();
  if (lower.endsWith('.cmd') || lower.endsWith('.bat') || lower.endsWith('.ps1')) return true;
  return false;
}
```

`shell: true`일 때 `args` 배열이 셸 메타문자 해석을 받으므로, `$(malicious)` 또는 `& del /f /s /q C:\` 같은 인자 주입이 가능합니다.

**권장 수정:** `.cmd`/`.bat` 명령 경로를 화이트리스트로 제한. `.ps1`은 `pwsh.exe -File <path>`를 `shell: false`로 사용.

### H5. `secrets.enc.json` 파일 퍼미션 미설정 — **[해결됨]** `mode: 0o600` (파일), `mode: 0o700` (디렉토리) 적용

**파일:** `src/main/config/secret-store.ts:173-176`

```typescript
private saveToDisk(data: SecretsFile): void {
  const dir = path.dirname(this.filePath);
  fs.mkdirSync(dir, { recursive: true });
  fs.writeFileSync(this.filePath, JSON.stringify(data, null, 2), 'utf-8');

```

`mode` 옵션 없이 파일 생성 → 기본 umask(보통 `0o644`) 적용 → **다른 로컬 사용자가 읽기 가능**.

**권장 수정:** `fs.writeFileSync(path, data, { mode: 0o600 })`, 디렉토리는 `{ mode: 0o700 }`.

## 양호한 부분 (잘 된 점)

- `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true` — Electron 보안 설정 올바름
- `safeStorage` 기반 SecretStore — OS 수준 암호화 의도 적절 (C4 제외)
- `randomBytes(32)`로 토큰 생성 — 256비트 엔트로피 확보
- 토큰을 SHA-256 해시로만 저장 — 평문 미보관
- 모든 SQL 쿼리 파라미터화 — SQL 인젝션 없음
- `execFile`에 `shell: false` — 셸 인젝션 방지 (H4의 Windows 예외 제외)
- `esc()` XSS 이스케이핑 — 웹 클라이언트 대부분 적용 (M1의 `error` 예외)
- `X-Forwarded-For` 무시 — IP 스푸핑 방지 (`extractIp`)
- 요청 본문 크기 제한 (`MAX_BODY_SIZE = 1MB`)
- `ensureEncryptionAvailable()` 가드 — 암호화 불가 시 비밀 저장 거부
